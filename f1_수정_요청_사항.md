## 2-1. Supabase 프로젝트 통합 (필수 — 현재 F1이 작동하지 않는 원인)

### 문제

F1은 개발 당시 **별도 Supabase 프로젝트** (`fhnlknegbzfobnmyeceh`)에서 작업했습니다. 하지만 팀 통합 프로젝트는 `bnfgbwwibnljynwgkgpt`입니다.

현재 상태:
- f0/F2/F3/F4/F5 → `bnfgbwwibnljynwgkgpt` (SUPABASE_URL)
- F1 asyncpg → `fhnlknegbzfobnmyeceh` (F1_DATABASE_URL) ← **다른 프로젝트!**

이로 인해 F1이 f0의 `pipeline_steps(step_key='0')` 데이터를 찾지 못합니다. f0가 데이터를 저장한 DB와 F1이 읽는 DB가 다릅니다.

### 해결 방법

F1의 테이블과 데이터를 팀 공용 프로젝트(`bnfgbwwibnljynwgkgpt`)로 옮겨야 합니다:

**1단계: 공용 프로젝트에서 F1 마이그레이션 SQL 실행**
- Supabase 대시보드 `bnfgbwwibnljynwgkgpt` > SQL Editor에서
- `backend/db/migrations/001~009*.sql` 순서대로 실행
- `backend/db/seed/01~05*.sql` 순서대로 실행

**2단계: F1_DATABASE_URL을 공용 프로젝트로 변경**
```
F1_DATABASE_URL=postgresql://postgres.bnfgbwwibnljynwgkgpt:<공용프로젝트비밀번호>@aws-1-ap-northeast-2.pooler.supabase.com:6543/postgres
```

**3단계: 기존 F1 프로젝트의 데이터 이관** (필요한 경우)
- `fhnlknegbzfobnmyeceh`의 `f1_allowed_ingredients`, `f1_forbidden_ingredients` 등에 커스텀 데이터가 있다면 공용 프로젝트로 복사
- 시드 SQL로 충분하면 이관 불필요

### 근본적 해결 (권장): asyncpg → supabase-py 전환

다른 기능(F2/F3/F4/F5)은 모두 `SUPABASE_URL` + `SUPABASE_SERVICE_KEY`로 supabase-py SDK를 사용합니다. DB 비밀번호나 별도 connection string이 필요 없습니다.

F1만 asyncpg 직접 연결을 사용해서 `F1_DATABASE_URL`(DB 비밀번호 포함)이 필요합니다. supabase-py로 전환하면:
- `F1_DATABASE_URL` 환경변수 불필요 (삭제 가능)
- `backend/db/connection.py` 불필요 (삭제 가능)
- `main.py`의 lifespan DB 풀 초기화 불필요 (삭제 가능)
- 다른 기능과 동일하게 `SUPABASE_URL`/`SUPABASE_SERVICE_KEY`만으로 동작

**변경 대상:**
- `backend/services/step1_ingredients_check.py` — asyncpg 쿼리 → supabase-py 쿼리
- `backend/services/step3_standards.py` — asyncpg 쿼리 → supabase-py 쿼리
- `backend/routers/feature1.py` — `get_conn_dep()` 의존성 제거
- `backend/routers/db_manager.py` — `get_conn_dep()` 의존성 제거

### 확인 방법

변경 후 아래 명령으로 검증:
```bash
cd backend
python -c "
import asyncio, os
from dotenv import load_dotenv
from pathlib import Path
load_dotenv(Path('.env'), override=True)
import asyncpg
async def test():
    pool = await asyncpg.create_pool(os.environ['F1_DATABASE_URL'], min_size=1, max_size=2, statement_cache_size=0)
    # f0 데이터가 보이는지 확인
    rows = await pool.fetch('SELECT step_key, status FROM pipeline_steps LIMIT 5')
    for r in rows: print(f'step_key={r[\"step_key\"]} status={r[\"status\"]}')
    await pool.close()
asyncio.run(test())
"
```
f0의 `step_key='0'` 데이터가 보이면 통합 완료.

## 5. f0 → F1 데이터 파이프라인 자동 연결 — PM이 임의로 구현한 부분

> **주의**: 이 연결은 PM이 병합 과정에서 임의로 작성한 것입니다.
> F1 담당자가 검토 후 마음에 안 들면 자유롭게 수정하세요.
> 아래에 수정 방법을 안내합니다.

### 현재 구현된 내용

`backend/routers/feature1.py`의 `/run` 엔드포인트에 f0 자동 연결이 추가되었습니다.

**동작 방식:**
- `POST .../feature/1/run`에 **body를 비워서** 보내면 → `pipeline_steps(step_key='0')`에서 자동 추출
- `POST .../feature/1/run`에 **body를 직접** 보내면 → 기존 방식 그대로 (하위 호환)

**자동 매핑 규칙:**

```
f0 ParsedResult                      →  F1 Ingredient
────────────────────────────────────────────────────
ingredients[].name                   →  name
ingredients[].ratio (str→float)      →  percentage
ingredients[].ins_number             →  ins
ingredients[].cas_number             →  cas
```

```
f0 process_info.process_codes        →  F1 ProcessConditions
────────────────────────────────────────────────────
가열 코드 (01,02,03,04,06,07,08,09,49,91,92)  →  is_heated=true
증류 코드 (35,41,42)                            →  is_distilled=true
발효 코드 (10,16,17,18)                         →  is_fermented=true
```

### 확인 사항

- 가열/증류/발효 코드 매핑이 정확한지 검토해주세요
- f0에 아직 없는 필드(`part`, `sub_ingredients`, `alcohol_percentage`)는 연결되지 않았음 → `f0_수정_요청_사항.md` 참조

### 수정하고 싶다면 — 여기만 보면 됩니다

변경해야 하는 파일은 **`backend/routers/feature1.py` 하나뿐**입니다.

PM이 추가한 코드는 아래 3개 함수와 2개 상수입니다:

```
위치: backend/routers/feature1.py

# 상수 (가열/증류/발효 공정 코드 셋)
_HEAT_CODES = {"01", "02", ...}     ← 코드 추가/제거 자유
_DISTILL_CODES = {"35", "41", "42"}
_FERMENT_CODES = {"10", "16", "17", "18"}

# 함수 1: f0 파싱 결과 조회
async def _fetch_f0_parsed_result(db, case_id) → dict
  └─ pipeline_steps(step_key='0')에서 ai_result 가져옴
  └─ 수정할 일 거의 없음 (단순 DB 조회)

# 함수 2: f0 원재료 → F1 Ingredient 변환
def _convert_f0_to_f1_ingredients(parsed) → list[Ingredient]
  └─ ⭐ 매핑 규칙을 바꾸고 싶으면 이 함수를 수정
  └─ 예: part, sub_ingredients 매핑 추가 시 여기에 로직 추가

# 함수 3: f0 공정코드 → F1 ProcessConditions 변환
def _convert_f0_to_process_conditions(parsed) → ProcessConditions
  └─ ⭐ 공정 변환 규칙을 바꾸고 싶으면 이 함수를 수정
  └─ 예: alcohol_percentage 추출 추가 시 여기에 로직 추가
```

**완전히 걷어내고 싶다면:**
1. 위 3개 함수 + 2개 상수를 삭제
2. `run_feature1_endpoint`에서 자동 추출 분기(`if not ingredients:` 블록)를 삭제
3. `Feature1RunRequest.ingredients`를 다시 `list[Ingredient]` (필수)로 변경

**매핑만 수정하고 싶다면:**
- `_convert_f0_to_f1_ingredients()` 함수 안의 `Ingredient(...)` 생성부만 수정
- `_convert_f0_to_process_conditions()` 함수 안의 코드 셋/로직만 수정
- 나머지는 건드릴 필요 없음

## 6. admin-db-manager 프론트엔드 미복사 — 필요 시 복원 가능

### 미복사 판단 이유

F1 원본에 있던 `features/admin-db-manager/` 폴더(원재료/기준치 DB 관리자 UI)를 병합에서 제외했습니다. 이유:

1. **백엔드(`db_manager.py`)는 복사 완료** — API는 작동하므로, Postman/curl로 관리 가능
2. **관리자 UI는 우선순위가 낮음** — 현재 단계에서는 파이프라인 기능 연결이 더 중요
3. **대시보드 네비게이션에 연결 안 됨** — 가져와도 접근할 수 있는 메뉴가 없어서 별도 작업 필요
4. **법령 DB 관리 UI(`/admin/law-update`)는 이미 별도 구현됨** — F1의 `admin-law-update/LawUpdatePage.tsx`와 역할 중복

### 빠진 파일 목록

```
병합완료된_원본/f1_이병찬/frontend/features/admin-db-manager/
├── IngredientsPage.tsx          — 허용/금지 원재료 CRUD UI
├── ThresholdsPage.tsx           — 첨가물 기준치 CRUD UI
├── api/dbManager.ts             — /api/v1/admin/db/{table} 호출 함수
├── components/OwnershipBadge.tsx — 소유자 표시 배지
├── components/RowActions.tsx    — 행 수정/삭제 버튼
├── hooks/useCurrentUser.ts     — 현재 로그인 사용자 정보
├── types.ts                    — 타입 정의
└── constants.ts                — API 경로 상수
```

### 복원하고 싶다면

1. 원본 폴더를 복사:
   ```bash
   cp -r 병합완료된_원본/f1_이병찬/frontend/features/admin-db-manager frontend/features/admin-db-manager
   ```

2. 페이지 라우트 생성:
   ```bash
   mkdir -p frontend/app/admin/db-manager/ingredients
   mkdir -p frontend/app/admin/db-manager/thresholds
   ```
   각 폴더에 `page.tsx`를 만들어서 해당 컴포넌트를 import

3. 대시보드 네비게이션에 링크 추가:
   `frontend/app/dashboard/page.tsx`의 검역관리 드롭다운 또는 별도 관리자 메뉴에 추가

4. 백엔드는 이미 준비됨 (`backend/routers/db_manager.py`가 `/api/v1/admin/db/{table}` 제공)

## [공통] 법령 DB 전처리 — HWPX/PDF 파일 처리 확인

법령 원본 파일이 HWPX(한글) 또는 PDF 형식으로 제공됩니다. 자기 기능의 법령 DB 전처리 스크립트가 **두 형식 모두 처리 가능한지** 확인해주세요.

**확인 사항:**
- PDF 파싱: `pdfplumber`, `PyMuPDF(fitz)`, `pdfminer` 등 사용 가능한지
- HWPX 파싱: `python-hwp`, 외부 파서 서비스, 또는 직접 XML 파싱 가능한지
- 전처리 스크립트에서 파일 확장자에 따라 분기 처리가 되어 있는지
- 법령 업데이트 시스템(`/admin/law-update`)에서 PDF/HWPX 업로드 시 자기 기능의 전처리가 정상 트리거되는지

**현재 상태:**
- f0: PDF(PyMuPDF) + HWPX(외부 파서 서비스) 지원
- F4: PDF(PyMuPDF) + HWPX 지원 (`preprocess_laws.py`)
- F1/F2/F3/F5: 각 담당자가 확인 필요
