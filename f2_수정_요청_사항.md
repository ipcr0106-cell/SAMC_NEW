## 1. Supabase 테이블 확인 (필수)

F2가 사용하는 테이블이 Supabase에 존재하는지 확인해주세요:

- `f2_food_type_classification` — 식품유형 분류 데이터 (대/중/소분류)
- `f2_required_documents` — 식품유형별 필요서류 매핑

이 테이블들은 Node.js 전처리 스크립트(`preprocessing/upload_foodcode5_full.js`, `upload_required_docs_f2.js`)로 생성됩니다. 우리 환경에서 아직 실행되지 않았다면 실행 필요.

## 2. .env 키 추가 (필수)

`backend/.env`에 아래 키를 추가해주세요:

```
F2_OPENAI_API_KEY=sk-...          # GPT-4o (식품유형 분류 LLM)
F2_PINECONE_API_KEY=pcsk_...      # Pinecone samc-a 인덱스
F2_PINECONE_INDEX=samc-a          # 인덱스명 (dim 1536, text-embedding-3-small)
```

기존 코드의 `OPENAI_API_KEY`, `PINECONE_API_KEY`, `PINECONE_INDEX`는 `F2_` prefix로 변경되었습니다.

## 4. 전처리 스크립트 — Node.js → Python 변환 요청

현재 전처리 스크립트(`backend/db/feature2/preprocessing/*.js`)가 모두 Node.js입니다. 통합 법령 업데이트 시스템(`admin_law_update.py`)에서 자동 트리거하려면 Python으로 변환이 필요합니다.

**대상 스크립트:**
- `upload_foodtype_chunks.js` → Pinecone 청크 업로드
- `upload_foodcode5_full.js` → 식품유형 분류 테이블 업로드
- `upload_required_docs_f2.js` → 필요서류 테이블 업로드
- 기타 `upload_*.js`

**당장 급하지 않음** — 현재는 데이터가 이미 올라가 있으면 동작합니다. 법령 개정 시 재전처리가 필요할 때 변환 필요.

## 5. 프론트엔드 API 경로 수정 필요

F2 프론트엔드(`frontend/features/feature2/FoodClassificationPage.tsx`)의 API 호출이 현재 파이프라인 구조와 맞지 않습니다.

**현재 (아람 원본):**
```typescript
apiPost('/step2', { ... })           // /step2 직접 호출
apiPost('/step2-verdict', { ... })   // /step2-verdict 직접 호출
apiFormPost('/step3', form)          // /step3 직접 호출
```

**변경 필요 (우리 파이프라인 구조):**
```typescript
apiPost(`/cases/${caseId}/pipeline/feature/2/run`, { ... })
apiClient.get(`/cases/${caseId}/pipeline/feature/2`)
apiClient.patch(`/cases/${caseId}/pipeline/feature/2`, { ... })
```

현재는 f0 stub 레이아웃 안에 컴포넌트를 넣어뒀으나, 내부 API 호출이 작동하지 않습니다. 아람님이 API 경로를 수정해주세요.

### 5-1. 파일 업로드 UI 제거 필요 (중요)

현재 `FoodClassificationPage.tsx`에 파일 업로드(STEP1) 기능이 포함되어 있습니다. 하지만 **우리 파이프라인에서는 f0 단계에서 이미 서류 업로드+OCR이 완료된 상태에서 F2가 시작**됩니다.

- F2 페이지에 파일 업로드 UI가 있으면 안 됩니다
- F2는 f0가 `documents.parsed_md`에 저장한 OCR 텍스트를 읽어서 분류만 수행해야 합니다
- 파일 업로드 관련 코드(`files`, `FileItem`, `isAnalyzing` 등)를 제거하고, "AI 분류 실행" 버튼만 남기세요

### 5-2. step2/step3 네이밍 재검토

현재 컴포넌트 내부에서 사용하는 단계 명칭이 혼란스럽습니다:
- "STEP 1" = 파일 업로드+분석 → **f0에서 이미 처리됨, 제거 대상**
- "STEP 2" = 식품유형 분류 → 이게 F2의 **핵심 결과**인데 "step2"라는 이름이 부적절
- "STEP 3" = 법령 근거 표시 → 분류 결과의 일부이지 별도 단계가 아님

F2 페이지는 **식품유형 분류 결과를 보여주는 한 화면**이어야 합니다. 내부 단계를 나눌 필요 없이:
1. "AI 분류 실행" 버튼 → 백엔드 `/feature/2/run` 호출
2. 분류 결과 표시 (대/중/소분류 + 법령 근거 + 필요서류)
3. 담당자 확인/수정 → `/feature/2` PATCH

## 6. f0 + F1 → F2 데이터 파이프라인 — PM이 임의로 구현한 부분

> **주의**: 이 연결은 PM이 병합 과정에서 임의로 작성한 것입니다.
> F2 담당자가 검토 후 마음에 안 들면 자유롭게 수정하세요.

### 변경 내용

기존: `documents.parsed_md` (OCR 원문 텍스트)만 읽었음
변경: **f0 + F1 결과를 합쳐서 구조화된 텍스트**를 GPT-4o에 전달

**연결 흐름:**
```
f0 (step_key='0') → 제품명, 수출국, 원재료 목록, 공정 코드/원문
                       ↓
F1 (step_key='1') → 수입판정 결과, 원재료별 허용/금지 상태, 수입불가 사유
                       ↓
F2: /feature/2/run → 위 데이터를 합쳐서 구조화된 텍스트 생성
                    → Pinecone RAG 검색
                    → GPT-4o 분류
                    → pipeline_steps(step_key='2')에 저장
```

### F2가 받는 구조화된 입력 (GPT-4o 프롬프트에 전달)

```
[f0에서 가져오는 것]
제품명: OO 위스키
수출국: 영국
원재료 목록:
  - 보리 (51%)
  - 맥아 (30%)
  - 정제수 (19%)
공정 코드: 35, 14, 15, 45
공정 원문: 맥아를 분쇄하고 물과 섞어 발효시킨 후 증류하여...

[F1에서 가져오는 것]
F1 수입판정: 수입 가능 (수입가능)
F1 원재료 판정:
  - 보리: allowed (식품공전 별표1)
  - 맥아: allowed (식품공전 별표1)
  - 정제수: allowed
```

f0이나 F1 결과가 아직 없으면 기존처럼 `documents.parsed_md` (OCR 원문)을 fallback으로 사용합니다.

### 수정하고 싶다면 — 여기만 보면 됩니다

`backend/routers/feature2.py`에서 PM이 추가한 함수 2개:

```
# 함수 1: pipeline_steps에서 결과 조회
def _fetch_pipeline_result(sb, case_id, step_key) → dict
  └─ 단순 DB 조회. 수정할 일 거의 없음

# 함수 2: f0 + F1 결과 → 구조화 텍스트 생성
def _build_enriched_text(sb, case_id, fallback_parsed_md) → str
  └─ ⭐ 입력 데이터 구성을 바꾸고 싶으면 이 함수를 수정
  └─ 어떤 필드를 GPT-4o에 전달할지 여기서 결정
```

**완전히 걷어내고 싶다면:**
1. 위 2개 함수 삭제
2. `run_feature2`에서 `_build_enriched_text` 호출 대신 기존 `selected_doc["parsed_md"]` 직접 사용
3. 기존 아람 원본과 동일하게 동작

### F2 출력값 (pipeline_steps step_key='2' → ai_result)

```
pipeline_steps (step_key='2')
├── status           "waiting_review" → "completed"
├── ai_result
│   ├── category_name      "주류"                    // 대분류 (식품군)
│   ├── category_no        "15"                      // 대분류 번호
│   ├── subcategory_name   "증류주류"                 // 중분류 (식품종)
│   ├── food_type          "위스키"                   // 소분류 ← 핵심 출력
│   ├── law_ref            "식품공전 제5장 15. 주류"   // 근거 법령
│   ├── reason             "맥아 51%, 증류공정..."     // 분류 근거 설명
│   ├── is_alcohol         true                      // 주류 여부
│   ├── required_docs[]                              // 필요서류 목록
│   │   ├── doc_name       "수입신고서"
│   │   ├── condition      null
│   │   ├── is_mandatory   true
│   │   ├── law_source     "수입식품안전관리특별법"
│   │   └── food_type      "위스키" 또는 null(공통)
│   └── source_doc         "원재료배합비율표.pdf"
├── final_result     ← 담당자 수정 후 최종값
└── edit_reason      ← 수정 사유
```

**다음 기능이 F2에서 가져가는 핵심 필드:**
- **F1** → `food_type` (Step 3 기준치 비교 시 식품유형별 기준 적용. 단 F1이 선행이므로 첫 실행 시에는 null)
- **F3** → `food_type`, `is_alcohol`, `required_docs` (필요서류 조회)
- **F4** → `food_type` (라벨 검토 시 식품유형별 표시기준 적용)

## 7. feature3.py → f2_required_docs.py 이름 변경됨

`feature3.py`라는 이름이 F3 담당자(유빈)와 충돌하므로 `f2_required_docs.py`로 변경했습니다.

- 기능: F2 결과(`food_type`)로 `f2_required_documents` 테이블을 조회하여 필요서류 반환
- F3 담당자가 본격 구현하면 이 파일은 대체될 예정
- API 경로: `GET/PATCH /cases/{case_id}/pipeline/feature/3`

**수정하고 싶다면:**
`backend/routers/f2_required_docs.py` 하나만 수정하면 됩니다.
