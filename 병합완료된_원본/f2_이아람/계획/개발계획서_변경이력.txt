        → 수입 불가 원료 발견 시 즉시 중단 가능 (이후 단계 낭비 없음)
      - 기준규격(허용 양)은 식품유형에 의존하므로 STEP2 이후가 맞음

[2026-04-10] v6 — 5개 기능 구조로 전면 재편
----------------------------------------------------------------

▣ 변경 배경
  팀원 5명 개발 + 1명 디자인 체제로 확정.
  기존 8단계(STEP1~6, STEP_A, STEP_B) 구조를
  회사 요구사항의 4개 항목에 맞춰 5개 기능으로 재구조화.

▣ 변경 사항

  [v6-1] 파이프라인 8단계 → 5개 기능으로 재편

    기존:
      STEP1(원재료허용) → STEP2(식품유형) → STEP_A(수입필요서류)
      → STEP3(기준규격) → STEP4(공정코드) → STEP5(성분재질)
      → STEP_B(수출국표시사항) → STEP6(한글라벨)

    변경 후:
      기능1(수입가능여부) → 기능2(식품유형분류)
      → 기능3(수입필요서류) → 기능4(수출국표시사항) → 기능5(한글표시사항)

    주요 통합:
      - 기능1 = 구 STEP1(원재료허용) + 구 STEP3(기준규격) 통합
      - 기능3 = 구 STEP_A 그대로
      - 기능4 = 구 STEP_B 그대로
      - 기능5 = 구 STEP6 그대로

  [v6-2] STEP4(공정코드 추천), STEP5(성분재질 코드 매칭) 제외

    이유: 5개 기능 구조에서 해당 기능 미포함
    제거된 파일:
      - services/step4_process_codes.py
      - services/step5_material_codes.py
      - utils/fuzzy_search.py
    제거된 DB 테이블:
      - process_codes (제조공정 코드 211개)
      - material_codes (성분재질 코드 약 12,000개)

  [v6-3] 서비스 파일 이름 변경 (step→feature 네이밍)

    step1_ingredients_check.py → feature1_import_check.py
    step2_food_type.py         → feature2_food_type.py
    step_a_required_docs.py    → feature3_required_docs.py
    step_b_foreign_label.py    → feature4_foreign_label.py

  [v6-4] 프론트엔드 페이지 경로 변경

    /cases/[id]/step1/ → /cases/[id]/feature1/
    /cases/[id]/step2/ → /cases/[id]/feature2/
    /cases/[id]/step_a/ → /cases/[id]/feature3/
    /cases/[id]/step_b/ → /cases/[id]/feature4/
    /cases/[id]/step6/ → /cases/[id]/feature5/

  [v6-5] 팀원 역할 분담 재배정

    병찬: 기능1 (수입 가능 여부 판정)

    이유: 회사 요구사항에 없는 기능
