## A. 개발계획서 v6와의 차이점

### A-1. `allowed_status` 3값 enum 추가 제안

| 항목 | 개발계획서 | 이 제안 |
|-----|----------|---------|
| `ingredient_list.is_allowed` | BOOLEAN (true/false) | 유지하되 `allowed_status` TEXT 컬럼 추가 ('permitted','restricted','prohibited') |
| 영향 | - | 세연(기능5 라벨)·기능3 담당도 `restricted` 분기 읽어야 할 수 있음 |

- [ ] 채팅 공지 템플릿: "ingredient_list에 allowed_status 컬럼 추가 제안. restricted 케이스(별표2) 표현을 위해 필요. 의견 필요."
- [ ] Reviewer: **전원** (schema.sql 수정)

### A-2. `forbidden_ingredients` 신규 테이블

| 항목 | 개발계획서 | 이 제안 |
|-----|----------|---------|
| 절대 금지 원료 처리 | `ingredient_list.is_allowed=false`로 혼재 | 별도 테이블로 분리 → Step 0 게이트 전용 |

- [ ] 채팅 공지 + PR → **전원 합의**
- [ ] 기능3(수입필요서류) 담당자에게 영향: "수입불가 판정 시 기능3 실행 안 함" 로직 공유

### A-3. `thresholds` 컬럼 보강 (conversion_factor 등)

| 항목 | 개발계획서 | 제안 |
|-----|----------|------|
| 환산계수 | `condition_text`에 문자열로 | `conversion_factor NUMERIC(8,6)` 별도 컬럼 |
| "불검출"·"음성" | NUMERIC `threshold_value` 저장 불가 | `threshold_text TEXT` 추가 |
| 중금속/미생물 구분 | 없음 | `standard_type TEXT` 추가 |

- [ ] 공유 파일(schema.sql) 수정 → 전원 합의

### A-4. Pydantic 모델 신규 추가

- `backend/models/judgment.py` **신규 파일** — 병찬 단독 소유 가능 (팀_컨벤션_룰 §2-1에 없는 신규)
- [ ] 채팅 공지만 하고 PR 진행 (전원 Reviewer 불필요)

### A-5. `backend/constants/gmo.py` 신규

- GMO 위험원료 하드코딩 상수
- 세연(기능5)의 `gmo_ingredients` 테이블과 역할 분리: 기능1 내부 판정용 상수 vs 기능5 라벨 문구 결정용 DB
- [ ] 세연과 경계 재확인 필요

## C. 타 담당자와의 경계 협의

### C-1. 아람 (기능 2 — 식품유형 분류)

- [ ] 출력 규격 확정: 기능2가 기능1에게 넘기는 `food_type` 문자열
  - 질문: "증류주"인가 "주류/증류주/일반증류주" 계층 표기인가?
  - 질문: `is_alcohol` 플래그를 명시적으로 포함할지?
- [ ] 기능2 **이전**에 기능1 Step 0(forbidden)과 Step 1(매칭)만 먼저 실행하는 부분 실행 케이스 필요
  - 개발계획서 §4-1은 기능1 → 기능2 순서인데, §4-1-2(기준치) 는 food_type을 요구. 어떻게 분리?
  - **제안**: Step 0·1만 기능2 앞에서 실행, Step 3(기준치)은 기능2 이후 기능1 재호출 형태

### C-2. 세연 (기능 5 — 한글표시사항)

- [ ] `allergy_list` 테이블 — 기능1 후처리의 알레르겐 교차검사는 **세연 영역**. 중복 구현 금지.
  - 제안: 기능1 결과에서 `ingredient_verdicts` + `sub_ingredients` 까지만 제공. 알레르기 판정은 기능5로 위임.
- [ ] `gmo_ingredients` 테이블 vs `backend/constants/gmo.py` 상수 — 역할 분리 합의
  - 제안: 상수는 기능1 내부 "GMO 서류 필요 여부" 판정용, DB는 기능5의 "GMO 라벨 문구" 생성용

### C-3. 기능3 담당자 (수입필요서류)

- [ ] 기능1 결과 중 `gmo_target_ingredients`, `required_documents_hint`가 기능3 입력이 될지 확정
  - 제안: 기능1은 힌트만 제공, 최종 서류 목록은 기능3이 `required_documents` 테이블 기반으로 산출
- [ ] `is_oem`, `has_organic_cert` 등 플래그의 출처: 정보입력 단계인가 제품 기본정보 업로드인가?

### C-4. 기능4 담당자 (수출국표시사항)

- [ ] 영향 적음. 기능4는 라벨 이미지 멀티모달이라 기능1 출력 의존도 낮음.
- [ ] 단, 기능1의 `ingredient_verdicts`를 참조해 교차검증할 수 있음 (라벨에 표기된 원재료 ↔ 실제 원재료 일치 여부).

### C-5. 공통 인프라 담당 (미확정)

- [ ] SSE 파이프라인 `routers/pipeline.py` 에서 기능1 호출점 합의
  - 이벤트 `feature_start`/`feature_result`/`feature_waiting`/`feature_confirmed` 사용
  - 기능1 내부에 Step 0/1/3 세 단계가 있는데 각각 SSE 이벤트 낼지, 기능1 전체 1개 이벤트로 묶을지?
- [ ] `pipeline_steps` 테이블 `step_key='1'` 1개 row로 저장 vs 여러 row 분할?
  - 개발계획서 §2 `pipeline_steps` 설계상 `(case_id, step_key)` UNIQUE → **1개 row + 내부 JSONB로 세부 단계 구분** 권장
