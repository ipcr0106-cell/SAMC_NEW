## 1. `ingredient_list` 보강 (병찬 담당)

### 현재 스키마 (개발계획서 v6)

```sql
CREATE TABLE ingredient_list (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name_ko         TEXT NOT NULL,
    name_scientific TEXT,
    name_en         TEXT,
    ins_number      TEXT,
    cas_number      TEXT,
    aliases         TEXT[],
    usage_part      TEXT,
    usage_condition TEXT,
    is_allowed      BOOLEAN DEFAULT true,
    law_source      TEXT,
    created_by      UUID REFERENCES auth.users(id)
);
```

### 보강 제안 (newsamc V002 대응)

`is_allowed BOOLEAN`은 **2값**이라 `restricted`(조건부) 표현 불가. 별표1/별표2/별표3 세 구분이 필요하므로 3값 enum 컬럼 추가 권장.

```sql
-- ▼ 추가 제안 (DEFAULT 있어서 컨벤션 §8 허용)
ALTER TABLE ingredient_list
  ADD COLUMN allowed_status TEXT NOT NULL DEFAULT 'permitted'
    CHECK (allowed_status IN ('permitted','restricted','prohibited'));

-- 마이그레이션: 기존 is_allowed 값 반영
UPDATE ingredient_list
   SET allowed_status = CASE
       WHEN is_allowed = false THEN 'prohibited'
       ELSE 'permitted'
   END;

-- conditions 컬럼 (usage_condition과 중복? → 통합 검토 필요)
-- 개발계획서의 usage_condition이 이미 있으므로 추가 안 해도 됨.
-- newsamc의 conditions == SAMC_NEW의 usage_condition 로 매핑.
```

> **매핑 결정**: newsamc `allowed_ingredients.conditions` → SAMC_NEW `ingredient_list.usage_condition` 그대로 사용. 별도 컬럼 추가 X.

### 인덱스 (newsamc V010 + V007)

```sql
-- 트라이그램 확장 활성화 (V001에서)
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- 기능 1 5단계 매칭의 5단계(퍼지)용 GIN 인덱스 3종
CREATE INDEX IF NOT EXISTS idx_ingredient_list_name_ko_trgm
  ON ingredient_list USING gin (name_ko gin_trgm_ops);

CREATE INDEX IF NOT EXISTS idx_ingredient_list_name_en_trgm
  ON ingredient_list USING gin (name_en gin_trgm_ops);

CREATE INDEX IF NOT EXISTS idx_ingredient_list_scientific_trgm
  ON ingredient_list USING gin (name_scientific gin_trgm_ops);

-- 개발계획서 v6 기존 인덱스 유지
-- idx_ingredient_list_name, idx_ingredient_list_scientific, idx_ingredient_list_ins, idx_ingredient_list_cas
```

### trgm 검색 RPC (newsamc V010 이식)

```sql
CREATE OR REPLACE FUNCTION public.search_ingredients_trgm(
  q text,
  k integer DEFAULT 30
)
RETURNS TABLE (
  id uuid,
  name_ko text,
  name_en text,
  name_scientific text,
  allowed_status text,
  usage_condition text,
  similarity real
)
LANGUAGE sql
STABLE
AS $$
  SELECT
    il.id,
    il.name_ko,
    il.name_en,
    il.name_scientific,
    il.allowed_status::text,
    il.usage_condition,
    GREATEST(
      COALESCE(similarity(il.name_ko, q), 0),
      COALESCE(similarity(il.name_en, q), 0),
      COALESCE(similarity(il.name_scientific, q), 0)
    )::real AS similarity
  FROM ingredient_list il
  WHERE
    il.name_ko % q
    OR (il.name_en IS NOT NULL AND il.name_en % q)
    OR (il.name_scientific IS NOT NULL AND il.name_scientific % q)
  ORDER BY similarity DESC
  LIMIT GREATEST(k, 1);
$$;
```

## 2. `forbidden_ingredients` 신규 테이블 (newsamc V013)

### 목적

식품유형과 무관하게 **수입 자체가 불가**한 원재료 마스터. 기능1의 Step 0 게이트에서 먼저 체크 → 적중 시 **즉시 파이프라인 중단**.

### 제안 DDL

```sql
CREATE TABLE IF NOT EXISTS forbidden_ingredients (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name_ko     TEXT NOT NULL,
  name_en     TEXT,
  aliases     TEXT[] NOT NULL DEFAULT '{}',
  category    TEXT NOT NULL
              CHECK (category IN ('drug','endangered','unauthorized','toxin','other')),
  law_source  TEXT,                 -- 개발계획서 컨벤션: law_source 네이밍 유지
  reason      TEXT,
  is_verified BOOLEAN DEFAULT false,
  created_by  UUID REFERENCES auth.users(id),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

COMMENT ON TABLE forbidden_ingredients IS
  '식품유형 무관 절대 금지 원재료. 기능1 수입 가능 판정 Step 0 게이트.';

CREATE INDEX idx_forbidden_ingredients_name_ko
  ON forbidden_ingredients (name_ko);

CREATE INDEX idx_forbidden_ingredients_name_ko_trgm
  ON forbidden_ingredients USING gin (name_ko gin_trgm_ops);
```

### category 의미

| category | 의미 |
|----------|------|
| `drug` | 마약류 관리법 금지 (대마, 양귀비, 코카잎 등) |
| `endangered` | CITES 부속서 I (호랑이 뼈, 천산갑 비늘 등) |
| `unauthorized` | 식약처 미허가 신소재 (카바카바, 에페드라) |
| `toxin` | 독성물질 (컴프리, 아코니틴 계열) |
| `other` | 기타 |

> 시드 데이터는 `06_초기시드_데이터.md` 참조 (newsamc V013 mock 8건).

## 3. `thresholds` 테이블 분화 vs 보강 결정

### 현재 스키마 (개발계획서 v6)

```sql
CREATE TABLE thresholds (
    id               UUID PRIMARY KEY,
    ingredient_name  TEXT NOT NULL,
    food_type        TEXT NOT NULL,
    threshold_value  NUMERIC NOT NULL,
    unit             TEXT NOT NULL,
    condition_text   TEXT,
    compound_group   TEXT,
    is_compound_limit BOOLEAN DEFAULT false,
    law_source       TEXT NOT NULL,
    law_article      TEXT,
    is_verified      BOOLEAN DEFAULT false,
    ...
);
```

### 문제점 대비 newsamc

| 항목 | 개발계획서 `thresholds` | newsamc `additive_limits` | newsamc `safety_standards` |
|------|------------------------|--------------------------|---------------------------|
| 첨가물 병용 그룹 | `compound_group` TEXT | `combined_group` TEXT + `combined_max` NUMERIC | — |
| 염→산 환산계수 | `condition_text` 문자열에 (리스크) | **`conversion_factor NUMERIC(8,6)`** 전용 컬럼 | — |
| 타르색소 총량 | 별도 레코드로? | `total_tar_limit NUMERIC` + `colorant_category`('tar'/'non-tar'/'natural') | — |
| "불검출"·"음성" 같은 비숫자 기준 | `threshold_value NUMERIC` (저장 불가) | — | **`max_limit TEXT`** (저장 가능) |
| 중금속/미생물/잔류농약 구분 | — | — | `standard_type`('microbe'/'heavy_metal'/'pesticide'/'contaminant') |

### 제안 옵션

**옵션 A (최소 변경)**: `thresholds`에 컬럼만 추가
```sql
ALTER TABLE thresholds
  ADD COLUMN IF NOT EXISTS conversion_factor NUMERIC(8,6),   -- 안식향산나트륨 0.847 등
  ADD COLUMN IF NOT EXISTS colorant_category TEXT
    CHECK (colorant_category IN ('tar','non-tar','natural') OR colorant_category IS NULL),
  ADD COLUMN IF NOT EXISTS standard_type TEXT,  -- 'additive'/'microbe'/'heavy_metal'/'pesticide'/'contaminant'
  ADD COLUMN IF NOT EXISTS threshold_text TEXT; -- "불검출","음성" 저장용 (NULL이면 숫자 기준 사용)
```

**옵션 B (분화)**: `additive_limits`·`safety_standards` 별도 테이블 신설
- 장점: 스키마가 깔끔하고 newsamc 로직을 그대로 이식
- 단점: 공유 파일 수정 + 마이그레이션 + 쿼리 복잡도 증가

> **추천**: **옵션 A**. 개발계획서 v6의 `thresholds` 구조를 최소 변경으로 확장. 데모 2주 기준 분화는 리스크.

## 4. 기능 1 판정 파이프라인과 DB 연결도

```
[입력] ingredients[{name, percentage, ins, cas, chemical_name}]
    │
    ▼
[Step 0] forbidden_ingredients 조회
    │   └── 적중 → 수입불가 (category/law_source/reason 반환) → 파이프라인 종료
    ▼
[Step 1] ingredient_list 5단계 매칭
    │   ├── 1차 정확명     → name_ko
    │   ├── 2차 INS        → ins_number
    │   ├── 3차 CAS        → cas_number
    │   ├── 4차 학명       → name_scientific
    │   └── 5차 퍼지(trgm) → search_ingredients_trgm() RPC
    │
    │   → allowed_status 에 따라:
    │     - prohibited → 수입불가
    │     - restricted → conditions 평가 (usage_condition)
    │     - permitted  → 다음 단계로
    │     - 미매칭      → unidentified (담당자 확인)
    ▼
[Step 2] thresholds 기준치 수치 비교 (food_type은 기능2 결과)
    │   ├── WHERE ingredient_name=$1 AND food_type=$2 AND is_verified=true
    │   ├── 단위 환산 (conversion_factor 활용)
    │   └── 개별 판정 + compound_group 합산 판정
    ▼
[출력] verdict + law_refs[]
```

## 5. 관련 마이그레이션 파일 순서 제안

개발계획서 스키마(v6)가 이미 `schema.sql`에 들어가 있다고 가정하고, 보강용 신규 마이그레이션 파일:

```
backend/db/migrations/
├── 001_enable_pg_trgm.sql
├── 002_add_ingredient_list_allowed_status.sql
├── 003_create_forbidden_ingredients.sql
├── 004_add_thresholds_columns.sql          (옵션 A 선택 시)
├── 005_ingredient_trgm_indexes.sql
└── 006_search_ingredients_trgm_rpc.sql
```

> **주의**: 개발계획서는 `backend/db/schema.sql` 단일 파일 전략이므로, 위 분할은 **제안**. 단일 파일 유지 시에는 `schema.sql` 하단에 섹션 추가.
