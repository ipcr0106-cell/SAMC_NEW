> 헌법: 개발계획서 §8 (DB 초기 데이터 입력 책임 - 병찬), Day 2 의무 (`ingredient_list` 200건)

## 1. 입력 목표

| 테이블 | Day 2 목표 | 최종 목표 | 출처 |
|--------|----------|----------|------|

## 2. 입력 순서 (의존성 고려)

```
Phase 2-A (D2 오전): ingredient_list 110건 (newsamc seed 이식)
    │
    ├── Phase 2-B (D2 오전): forbidden_ingredients 8건 (mock)
    │
    └── Phase 2-C (D2 오후): thresholds 핵심 20건 수동 입력
           │
```

Phase 2-A·B·C는 병렬 가능, D는 기능1 구현과 병행.

## 3. Phase 2-A: ingredient_list 110건

### 3-1. 파일 구조

```
backend/db/seed/
├── 01_ingredient_permitted.sql      # 별표1: 85건
├── 02_ingredient_restricted.sql     # 별표2: 10건
├── 03_ingredient_prohibited.sql     # 별표3: 15건
```

### 3-2. 작성 규칙 (팀컨벤션 §8 준수)

- `law_source` 컬럼 **반드시 기재** ("식품공전 [별표1]" 등)
- `is_verified=true` (검수 완료 시드)
- `created_by=NULL` (시스템 초기 데이터로 표시, DB 관리 UI에서 수정/삭제 불가)
- ON CONFLICT DO NOTHING 사용 → 재실행 안전

### 3-3. 변환 매핑 (newsamc → SAMC_NEW)

newsamc `allowed_ingredients` 컬럼 → SAMC_NEW `ingredient_list` 컬럼

| newsamc | SAMC_NEW | 비고 |
|---------|----------|-----|
| `name_ko` | `name_ko` | 그대로 |
| `name_en` | `name_en` | 그대로 |
| `scientific_name` | `name_scientific` | 이름 변경 |
| `allowed_status` | `allowed_status` (§01 PR-02 추가) | 그대로 |
| `conditions` | `usage_condition` | 이름 변경 |
| `ins_number`, `cas_number` | 동일 | NULL 허용 |
| — | `is_allowed` | `allowed_status != 'prohibited'` 로 채움 |
| — | `law_source` | 수동 추가 ("식품공전 [별표1]" 등) |
| — | `created_by` | NULL |
| — | `aliases` | 필요 시 추가 |

### 3-4. 변환 스크립트 (일회용)

newsamc의 `supabase/seed/05_allowed_ingredients.sql`을 읽어 변환 SQL을 생성하는 Python 스크립트를 `backend/scripts/convert_seed.py`에 작성. **반복 사용 안 하므로 간단하게**.

```python
# backend/scripts/convert_seed.py (일회용)
import re, sys

SRC = r"C:\GITHUB\newsamc\supabase\seed\05_allowed_ingredients.sql"
DST_MAP = {
    "permitted":  "backend/db/seed/01_ingredient_permitted.sql",
    "restricted": "backend/db/seed/02_ingredient_restricted.sql",
    "prohibited": "backend/db/seed/03_ingredient_prohibited.sql",
}
LAW_SOURCE_MAP = {
    "permitted":  "식품공전 [별표1]",
    "restricted": "식품공전 [별표2]",
    "prohibited": "식품공전 [별표3]",
}

# 정규식으로 newsamc 5컬럼 INSERT → SAMC_NEW 9컬럼 INSERT 변환
# (name_ko, name_en, scientific, status, conditions)
#   →
# (name_ko, name_en, name_scientific, allowed_status, is_allowed,
#  usage_condition, law_source, is_verified, created_by)
# ... 상세 구현 생략, 변환 후 DST_MAP 경로에 저장
```

### 3-5. 검증

```sql
SELECT allowed_status, COUNT(*) FROM ingredient_list GROUP BY allowed_status;
-- permitted 85, restricted 10, prohibited 15

SELECT COUNT(*) FROM ingredient_list WHERE law_source IS NULL;
-- 0이어야 함 (law_source 누락 방지)

SELECT COUNT(*) FROM ingredient_list WHERE allowed_status='restricted' AND usage_condition IS NULL;
-- 0이어야 함 (restricted 조건 누락 방지)
```

5. 수정 필요 시 UPDATE 쿼리로 즉시 반영

## 6. Phase 2-D: LLM 추출 확장 (Day 3+)

### 6-1. 흐름

```
[HWP/PDF 법령 파일]
    ↓ (관리자 UI: /admin/law-update)
[parser-service] kordoc 변환 → 마크다운
    ↓
[cleaner.py] 정제
    ↓
[chunker.py] 조문·표 단위 청킹
    ↓ (두 갈래)
[Pinecone] 벡터 저장 (RAG용)
[law_extractor.py] Claude API 수치 추출 (JSON)
    ↓
[관리자 UI 미리보기] — is_verified=false 상태로 저장
    ↓ (담당자 수동 검수)
[thresholds UPDATE] is_verified=true
```

### 6-2. 데모 전까지 현실 목표

- LLM 추출이 완벽할 필요 없음 (개발계획서 리스크 3 Plan B)
- **핵심 20건 수동 입력**이 Primary, LLM 추출은 Secondary
- 데모에서는 수동 입력 20건만 보여주고 "LLM 추출 파이프라인 구축됨" 정도로 시연

## 7. 병찬 개인 관리 페이지 활용

개발계획서 §3-5 "DB 직접 관리" 프론트엔드와 연결:
- `/admin/db-manager/ingredients` → `created_by=내 UUID`인 행만 수정/삭제
- 시드는 `created_by=NULL`이라 "시스템 데이터" 표기, 수정 불가
- 병찬이 개인적으로 테스트 데이터 추가 시 `created_by=본인`으로 들어가 구분됨
