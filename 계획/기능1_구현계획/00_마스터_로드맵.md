## 1. 목표 재확인

### 1-1. 비즈니스 목표
정보입력 단계의 **원재료 목록 + 배합비율 + INS/CAS/화학성분명**을 입력받아, 한국 식품법령 기준으로 **수입 가능 여부를 판정**하고 **법령 근거**를 제시한다.

### 1-2. 기능 경계 (개발계획서 §4-1 + §4-1-2 통합)
기능 1 = **Step 0(forbidden) + Step 1(성분 매칭) + Step 3(기준치 검증)** 을 하나로 묶은 단위.
- Step 0: `forbidden_ingredients` 조회 → 적중 시 **파이프라인 즉시 중단**
- Step 1: 5단계 매칭 + 복합원재료 재귀 + 조건부(restricted) 평가
- Step 3: `thresholds` 기준치 수치 비교 (일반식품 + 주류 분기, `food_type`은 기능2에서 받음)

### 1-3. 비목표 (타 담당자 영역)
- ❌ 알레르기 감지 → 세연(기능5)
- ❌ GMO 라벨 문구 → 세연(기능5)
- ❌ 필요서류 목록 산출 → 기능3 담당
- ❌ 식품유형 분류 → 아람(기능2)
- ❌ SSE 파이프라인 오케스트레이션 → 공통 인프라 담당

## 3. Go / No-Go 체크포인트

각 Phase 종료 시 아래 조건을 만족해야 다음 Phase 진입.

| 체크포인트 | Go 조건 | No-Go 대응 |
|-----------|--------|-----------|
| Phase 0 종료 | schema 변경 전원 합의 완료 + 경계 이슈 없음 | 협의 미완 시 Day 2로 이월, Phase 1은 "옵션 A 최소 변경"만 우선 PR |
| Phase 1 종료 | schema.sql 머지 + Supabase 실행 성공 + 쿼리 샘플 통과 | 실행 실패 시 롤백 SQL 준비 (§01 문서 참조) |
| Phase 2 종료 | `ingredient_list` ≥ 200건, `is_verified=true`, `search_ingredients_trgm('쌀')` 작동 | 데이터 부족 시 LLM 추출 → 검수 루프 기동 |
| Phase 3 종료 | Step 0·1·1-A·1-B·Step 3 단위 테스트 통과 | 실패 시 블로킹 원인만 픽스 (리팩토링 금지, 리스크 5 원칙) |
| Phase 6 종료 | step1·step3·admin 페이지 렌더 + lib/api.ts 연결 | 디자이너 스타일 미지원 시 최소 스타일로 진행 |
| Phase 7 종료 | 3종 샘플(통과/금지/초과) E2E 각 1회 성공 | 1회라도 실패 시 Phase 8 보류 |

## 7. 진행 원칙 (읽기 싫으면 이것만)

1. **개발계획서 네이밍 우선** — 충돌 시 팀 컨벤션 헌법 승리
2. **공유 파일은 채팅 공지 → PR → 전원 Reviewer** — 단독 커밋 금지
3. **`is_verified=true`만 판정에 사용** — 리스크 3 강제
4. **LLM은 수치 비교 안 함** — DB 조회 + Python `<`, `>` 만 사용
5. **Step 0(forbidden) 적중 = 즉시 파이프라인 종료** — 이후 단계 실행 X
6. **에스컬레이션 우선** — 애매하면 담당자에게 넘기기 (ambiguous → 수동 확인)
7. **테스트 없이 완료 주장 금지** — 최소 8개 시나리오 통과 후 "완료" 보고
