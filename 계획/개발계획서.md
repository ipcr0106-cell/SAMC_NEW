### 1-2. 서비스 분리 이유

세 서비스를 분리하는 이유는 단 하나다. kordoc이 Node.js 전용이고, 백엔드가 Python FastAPI이기 때문이다. Python에서 HWP를 제대로 파싱하는 라이브러리가 없다. 마이크로서비스 분리가 2주 데모 기준 가장 현실적인 선택이다 (자세한 이유는 섹션 6 참조).

## 2. DB 스키마 (PostgreSQL / Supabase)

```sql
-- ================================================================
-- SAMC 수입식품 검역 AI 플랫폼 - DB 스키마
-- Supabase(PostgreSQL) 기준. supabase.com 대시보드 SQL 에디터에서 실행.
-- ================================================================

-- 건(Case) 목록: 검역 처리 단위
CREATE TABLE cases (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    product_name    TEXT NOT NULL,                        -- 제품명 (예: "FJ 캡 프론티어 위스키")
    importer_name   TEXT NOT NULL,                        -- 수입업체명
    status          TEXT NOT NULL DEFAULT 'processing'
                    CHECK (status IN ('processing','completed','on_hold','error')),
                    -- 처리중/완료/보류/오류
    current_step    TEXT DEFAULT '0',                     -- 현재 진행 단계 (0=파싱, 1~6=숫자단계, A/B=신규단계)
    parent_case_id  UUID REFERENCES cases(id),            -- 재수입 건의 원본 건 UUID (신규 건이면 NULL)
    created_by      UUID REFERENCES auth.users(id),       -- Supabase Auth 사용자
    locked_by       UUID REFERENCES auth.users(id),       -- 편집 잠금 (동시접속 처리)
    locked_at       TIMESTAMPTZ,                          -- 잠금 시각 (10분 타임아웃)
    created_at      TIMESTAMPTZ DEFAULT now(),
    updated_at      TIMESTAMPTZ DEFAULT now()
);

-- 업로드된 서류 목록
CREATE TABLE documents (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    case_id     UUID NOT NULL REFERENCES cases(id) ON DELETE CASCADE,
    doc_type    TEXT NOT NULL
                CHECK (doc_type IN ('ingredients','process','msds','material','other')),
                -- 원재료배합비/제조공정도/MSDS/성분재질서류/기타
    file_name   TEXT NOT NULL,                           -- 원본 파일명
    storage_path TEXT NOT NULL,                          -- Supabase Storage 경로
    mime_type   TEXT,                                    -- application/pdf, image/jpeg 등
    parsed_md   TEXT,                                    -- kordoc/Claude 파싱 결과 (마크다운). 대용량(1MB+)이면 Supabase Storage에 별도 저장 후 경로만 기록.
    is_verified BOOLEAN DEFAULT false,                   -- 담당자 확인 완료 여부
    created_at  TIMESTAMPTZ DEFAULT now()
);

-- 8단계 파이프라인 각 단계 결과 저장
-- step_key: '1'~'6' 숫자 단계, 'A'=수입필요서류, 'B'=수출국표시사항검토
CREATE TABLE pipeline_steps (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    case_id         UUID NOT NULL REFERENCES cases(id) ON DELETE CASCADE,
    step_key        TEXT NOT NULL,                        -- '1','2','A','3','4','5','B','6'
    step_name       TEXT NOT NULL,
                    -- 'ingredients_check'/'food_type'/'required_docs'/'standards'/
                    -- 'process_codes'/'material_codes'/'foreign_label'/'label'
    status          TEXT NOT NULL DEFAULT 'pending'
                    CHECK (status IN ('pending','running','waiting_review','completed','error')),
    ai_result       JSONB,                               -- AI 원본 출력 (JSON 구조화)
    final_result    JSONB,                               -- 담당자 확인/수정 후 최종값
    edited_by       UUID REFERENCES auth.users(id),     -- 수정한 담당자
    edit_reason     TEXT,                                -- 수정 사유 (식약처 소명용)
    validation_result JSONB,                             -- 법령 정합성 검토 결과 (충돌·의존 분석)
    created_at      TIMESTAMPTZ DEFAULT now(),
    updated_at      TIMESTAMPTZ DEFAULT now(),
    UNIQUE (case_id, step_key)
);

-- 기준규격 임계값 DB (STEP3에서 수치 비교용)
-- LLM이 법령에서 추출 → 사람 검수 후 여기 저장 → 코드가 조회해서 비교
CREATE TABLE thresholds (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    ingredient_name  TEXT NOT NULL,                      -- 성분명 (예: "안식향산나트륨")
    food_type        TEXT NOT NULL,                      -- 식품유형 (예: "과채음료")
    threshold_value  NUMERIC NOT NULL,                   -- 기준값 (예: 0.6)
    unit             TEXT NOT NULL,                      -- 단위 (예: "g/kg", "%", "mg/L")
    condition_text   TEXT,                               -- 조건 설명 (예: "가열제품에 한함")
    compound_group   TEXT,                               -- 복합 합산 그룹명 (예: "타르색소합산"). NULL이면 개별 판정.
    is_compound_limit BOOLEAN DEFAULT false,             -- true면 compound_group 합산값으로 그룹 기준치와 비교
    law_source       TEXT NOT NULL,                      -- 출처 법령명
    law_article      TEXT,                               -- 조항 번호 (예: "제3조 제1항")
    is_verified      BOOLEAN DEFAULT false,              -- 사람 검수 완료 여부 (필수). false이면 STEP3에서 사용 불가.
    extracted_at     TIMESTAMPTZ DEFAULT now(),          -- LLM 추출 시각
    verified_by      UUID REFERENCES auth.users(id),    -- 검수자
    verified_at      TIMESTAMPTZ,
    created_by       UUID REFERENCES auth.users(id)     -- 추가한 사용자 (NULL이면 시스템 초기 데이터)
);

-- process_codes (제조공정 코드), material_codes (성분재질 코드) 테이블은
-- 5개 기능 구조에서 제외됨 (기능4/5에서 공정코드·성분코드 매칭 기능 미포함)

-- 식품원료목록 - 포지티브 리스트 (STEP1 원재료 허용 여부 판정용)
CREATE TABLE ingredient_list (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name_ko         TEXT NOT NULL,                       -- 한국어 원료명
    name_scientific TEXT,                                -- 학명 (예: "Agave tequilana")
    name_en         TEXT,                                -- 영문명
    ins_number      TEXT,                                -- INS 번호 (예: "468") — 식품첨가물공전 IV.품목별 성분규격 출처
    cas_number      TEXT,                                -- CAS 번호 (예: "74811-65-7")
    aliases         TEXT[],                              -- 동의어 배열 (예: {"Croscarmellose sodium","Cross-linked CMC"})
    usage_part      TEXT,                                -- 사용 부위 (잎, 줄기, 열매 등)
    usage_condition TEXT,                                -- 사용 조건/제한
    is_allowed      BOOLEAN DEFAULT true,                -- 허용 여부
    law_source      TEXT,                                -- 근거 법령
    created_by      UUID REFERENCES auth.users(id)       -- 추가한 사용자 (NULL이면 시스템 초기 데이터)
);

-- 법령 개정 이력 및 이메일 알림 로그
CREATE TABLE law_alerts (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    law_name        TEXT NOT NULL,                       -- 개정된 법령명
    change_summary  TEXT,                                -- 변경 내용 요약 (예: "안식향산나트륨 0.6→0.5g/kg")
    affected_steps  INTEGER[],                           -- 영향받는 파이프라인 단계 번호
    file_uploaded_by UUID REFERENCES auth.users(id),    -- 업로드한 관리자
    email_sent      BOOLEAN DEFAULT false,               -- 이메일 발송 여부
    email_sent_at   TIMESTAMPTZ,
    created_at      TIMESTAMPTZ DEFAULT now()
);

-- 담당자 피드백 로그 (AI 결과 수정 이력 학습용)
CREATE TABLE feedback_logs (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    case_id         UUID REFERENCES cases(id),
    step_key        TEXT,                                -- 수정된 단계 ('1','2','A','3','4','5','B','6')
    ai_suggestion   JSONB,                               -- AI 원본 추천값
    final_value     JSONB,                               -- 담당자 최종 선택값
    edit_reason     TEXT,                                -- 수정 사유
    user_id         UUID REFERENCES auth.users(id),
    created_at      TIMESTAMPTZ DEFAULT now()
);

-- 수입 필요서류 목록 (STEP A에서 식품유형별 조회용)
-- 출처: DB_최신/6_가이드라인/수입신고시_제출하여야_하는_구비서류_목록(2026.2.5.).xlsx
CREATE TABLE required_documents (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    food_type       TEXT,                                -- 식품유형 (NULL이면 모든 식품 공통 서류)
    condition       TEXT,                                -- 추가 조건 (예: 'OEM', '친환경인증', '주류')
    doc_name        TEXT NOT NULL,                       -- 서류명 (예: "제조국 정부기관 발행 위생증명서")
    doc_description TEXT,                                -- 상세 설명 및 발급처
    is_mandatory    BOOLEAN DEFAULT true,                -- 필수 여부 (false이면 조건부 필요)
    law_source      TEXT,                                -- 근거 출처 (예: "수입식품안전관리특별법 제20조")
    created_by      UUID REFERENCES auth.users(id)       -- 추가한 사용자 (NULL이면 시스템 초기 데이터)
);

-- 인덱스 (성능)
CREATE INDEX idx_cases_status ON cases(status);
CREATE INDEX idx_cases_created_by ON cases(created_by);
CREATE INDEX idx_pipeline_steps_case_id ON pipeline_steps(case_id);
CREATE INDEX idx_thresholds_ingredient ON thresholds(ingredient_name, food_type);
CREATE INDEX idx_thresholds_compound ON thresholds(compound_group) WHERE compound_group IS NOT NULL;
CREATE INDEX idx_ingredient_list_name ON ingredient_list(name_ko);
CREATE INDEX idx_ingredient_list_scientific ON ingredient_list(name_scientific);
CREATE INDEX idx_ingredient_list_ins ON ingredient_list(ins_number) WHERE ins_number IS NOT NULL;
CREATE INDEX idx_ingredient_list_cas ON ingredient_list(cas_number) WHERE cas_number IS NOT NULL;
CREATE INDEX idx_feedback_logs_feature ON feedback_logs(feature_key);
CREATE INDEX idx_required_documents_food_type ON required_documents(food_type);

-- 업데이트 시각 자동 갱신 트리거
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = now();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_cases_updated_at
    BEFORE UPDATE ON cases
    FOR EACH ROW EXECUTE FUNCTION update_updated_at();

CREATE TRIGGER trigger_pipeline_steps_updated_at
    BEFORE UPDATE ON pipeline_steps
    FOR EACH ROW EXECUTE FUNCTION update_updated_at();
```

## 3. API 엔드포인트 전체 목록

FastAPI 기준. 모든 엔드포인트는 `/api/v1` prefix. 인증은 Supabase JWT Bearer 토큰.

### 3-1. 건(Case) 관리

| Method | 경로 | 설명 |

#### 실제 법령 파일에서 발견된 구조적 문제 2가지

식품공전 실제 마크다운 파일(`식품공전_2__제6_제7.md`)을 분석한 결과, 청킹 전에 반드시 처리해야 할 문제가 두 가지 발견됐다.

**문제 A — 줄 앞이 잘린 텍스트 (kordoc 변환 아티팩트)**

원본 HWP가 2단 레이아웃이거나 페이지 경계에서 잘리면, 변환 후 마크다운에 아래처럼 줄 앞이 잘린 텍스트가 나타난다:

```
# 원본 HWP에서 의도한 문장:
"제8. 일반시험법 4. 미생물시험법 4.8 대장균에 따라 시험한다."

# kordoc 변환 후 실제 결과:
"험법 4. 미생물시험법 4.8 대장균에 따라 시험한다."
"치에 여과막의 격자가 그려진 면을 위로 향하게..."
```

이 상태로 벡터 DB에 넣으면 "대장균 시험방법"으로 검색해도 해당 청크가 제대로 검색되지 않는다.

**해결책 — cleaner.py 정제 단계 추가**:

```python
# utils/cleaner.py
import anthropic

async def clean_law_markdown(raw_markdown: str) -> str:
    """
    kordoc 변환 후 잘린 줄, 이상 문자를 Claude로 정제.
    전체 파일을 한 번에 보내지 않고, 페이지/섹션 단위로 나눠서 처리.
    """
    # 1단계: 규칙 기반 빠른 정제 (90% 케이스 처리)
    import re
    # 조문 번호로 시작하지 않는 줄 중, 직전 줄과 연결 가능한 것 병합
    lines = raw_markdown.split("\n")
    merged = []
    for i, line in enumerate(lines):
        # 앞이 잘린 줄 패턴: 소문자/숫자로 시작 + 직전 줄이 완전한 문장이 아님
        if (merged and line and
            not line.startswith(("#", "|", "제", "1", "2", "3", "4", "5", "가", "나", "※", "-")) and
            not merged[-1].endswith((".", "다.", "한다.", "된다.", "한다", "한다."))):
            merged[-1] = merged[-1] + line  # 직전 줄에 합침
        else:
            merged.append(line)

    cleaned = "\n".join(merged)

    # 2단계: 여전히 이상한 부분은 Claude로 처리 (섹션 단위)
    # 파일 전체를 Claude에 넣으면 토큰 초과 → 제N조 단위로 분할해서 처리
    return cleaned
```

**문제 B — 표(Table)가 조문 중간에 끼어 있음**

법령 파일에는 아래처럼 표가 설명 텍스트 + 표 + 각주(※) 구조로 되어 있다:

```
[설명 텍스트] "...검체채취결정표에 따라 검체를 채취하며..."
[표]
| 검사대상 크기(kg) | 검체채취 지점수 | 시험검체수 |
| ∼ 5,000 미만 | 2 | 1 |
...
[각주] ※ 25,000 kg 이상 100,000 kg 미만인 경우에는 4곳 이상에서...
```

조문 경계에서만 자르면 표와 앞뒤 설명이 분리되어 표의 의미를 잃는다.

**해결책 — 표 컨텍스트 보존 청킹 규칙**:

```python
# utils/chunker.py
def chunk_law_markdown(markdown_text: str) -> list[dict]:
    """
    조문 단위 청킹 + 표 컨텍스트 보존.
    
    규칙:
    1. 조문(제N조, 제N항, N.) 경계에서 분할 — 기본 원칙
    2. 단서 조항("단, ...") 은 직전 조문과 합쳐서 하나의 청크로 — 기존 원칙
    3. 표(|로 시작하는 줄)는 절대 중간에서 자르지 않음
    4. 표가 등장하면 → 표 앞 2문장 + 표 전체 + 표 뒤 각주(※)를 하나의 청크로 묶음
    5. 표 청크의 메타데이터에 {type: "table", parent_article: "제7조 3항"} 추가
    """
    chunks = []
    lines = markdown_text.split("\n")
    current_chunk_lines = []
    current_article = ""
    in_table = False
    table_context_lines = []  # 표 앞 2문장 버퍼

    for i, line in enumerate(lines):
        is_table_line = line.strip().startswith("|")
        is_footnote = line.strip().startswith("※")

        # 표 시작 감지
        if is_table_line and not in_table:
            in_table = True
            # 표 앞 2문장을 표 청크에 포함시키기 위해 현재 청크에서 분리
            if len(current_chunk_lines) >= 2:
                pre_table_context = current_chunk_lines[-2:]  # 직전 2줄
                current_chunk_lines = current_chunk_lines[:-2]
                # 현재까지의 내용을 청크로 저장
                if current_chunk_lines:
                    chunks.append({"text": "\n".join(current_chunk_lines),
                                   "article": current_article, "type": "text"})
                current_chunk_lines = pre_table_context  # 표 청크 시작

        # 표 끝 감지 (표 줄이 아닌 줄이 나왔을 때)
        if in_table and not is_table_line:
            in_table = False
            if is_footnote:
                current_chunk_lines.append(line)  # 각주는 표 청크에 포함
                # 표 청크 완성
                chunks.append({"text": "\n".join(current_chunk_lines),
                               "article": current_article, "type": "table"})
                current_chunk_lines = []
            else:
                # 각주 없이 끝나는 경우
                chunks.append({"text": "\n".join(current_chunk_lines),
                               "article": current_article, "type": "table"})
                current_chunk_lines = [line]
            continue

        # 조문 경계 감지 (표 안에서는 적용 안 함)
        article_pattern = re.match(r'^(제\d+조|제\d+항|\d+\.\s)', line)
        if article_pattern and not in_table and current_chunk_lines:
            # 단서 조항이면 자르지 않음
            if not line.strip().startswith("단,"):
                chunks.append({"text": "\n".join(current_chunk_lines),
                               "article": current_article, "type": "text"})
                current_chunk_lines = []
                current_article = article_pattern.group()

        current_chunk_lines.append(line)

    # 마지막 청크
    if current_chunk_lines:
        chunks.append({"text": "\n".join(current_chunk_lines),
                       "article": current_article, "type": "text"})

    return chunks
```

### 4-1-2. 기능 1 세부: 기준규격 적합성 검토 (기능 1에 통합)

> 기능 1(수입 가능 여부 판정)의 두 번째 파트. 원재료 허용 확인 후 기준규격 수치를 비교한다.  
> **핵심 원칙**: LLM이 수치 계산 안 함. DB에서 기준치 조회 → Python 코드가 비교.

```python
async def check_standards(
    ingredients: list[dict],  # [{"name": "안식향산나트륨", "value": 0.5, "unit": "g/kg"}]
    food_type: str,
    process_conditions: dict  # {"is_heated": True, "is_fermented": False}
) -> dict:
    results = []

    for ingredient in ingredients:
        # DB에서 기준치 조회 (is_verified=true 인 것만)
        threshold = await db.fetchrow("""
            SELECT * FROM thresholds
            WHERE ingredient_name = $1
              AND food_type = $2
              AND is_verified = true
            ORDER BY verified_at DESC LIMIT 1
        """, ingredient["name"], food_type)

        if not threshold:
            results.append({
                "ingredient": ingredient["name"],
                "status": "no_threshold",
                "message": f"{food_type}의 {ingredient['name']} 기준치가 DB에 없음 — 담당자 직접 확인 필요"
            })
            continue

        # 조건 체크 (예: 가열제품 한정)
        if threshold.condition_text and "가열" in threshold.condition_text:
            if not process_conditions.get("is_heated"):
                results.append({
                    "ingredient": ingredient["name"],
                    "status": "condition_not_met",
                    "message": f"이 기준은 가열제품에만 적용. 비가열 제품이면 다른 기준 적용 확인 필요",
                    "law_ref": threshold.law_article
                })
                continue

        # 단위 변환 (필요시)
        actual_value = convert_unit(ingredient["value"], ingredient["unit"], threshold.unit)

        # 수치 비교 (LLM 아님, 순수 Python)
        if actual_value > threshold.threshold_value:
            results.append({
                "ingredient": ingredient["name"],
                "status": "exceeded",
                "actual": actual_value,
                "limit": threshold.threshold_value,
                "unit": threshold.unit,
                "law_ref": f"{threshold.law_source} {threshold.law_article}"
            })
        else:
            results.append({
                "ingredient": ingredient["name"],
                "status": "passed",
                "actual": actual_value,
                "limit": threshold.threshold_value,
                "unit": threshold.unit
            })

    # 복합 합산 판정 (착색료 등 compound_group이 있는 성분들)
    compound_groups = {}
    for r in results:
        if r.get("status") in ("passed", "exceeded"):
            # 해당 성분의 compound_group 조회
            thr = await db.fetchrow(
                "SELECT compound_group, is_compound_limit FROM thresholds "
                "WHERE ingredient_name = $1 AND food_type = $2 AND is_verified = true",
                r["ingredient"], food_type
            )
            if thr and thr["compound_group"]:
                group = thr["compound_group"]
                if group not in compound_groups:
                    compound_groups[group] = {"sum": 0, "members": []}
                compound_groups[group]["sum"] += r.get("actual", 0)
                compound_groups[group]["members"].append(r["ingredient"])

    compound_results = []
    for group_name, group_data in compound_groups.items():
        # 그룹 합산 기준치 조회 (is_compound_limit=true인 레코드)
        group_threshold = await db.fetchrow(
            "SELECT threshold_value, unit, law_article, law_source FROM thresholds "
            "WHERE compound_group = $1 AND is_compound_limit = true AND food_type = $2 AND is_verified = true",
            group_name, food_type
        )
        if group_threshold:
            total = group_data["sum"]
            limit = group_threshold["threshold_value"]
            compound_results.append({
                "group": group_name,
                "members": group_data["members"],
                "total": total,
                "limit": limit,
                "unit": group_threshold["unit"],
                "status": "exceeded" if total > limit else "passed",
                "message": f"{group_name} 합산 실제 {total} > 기준 {limit} {group_threshold['unit']} — 합산 초과"
                           if total > limit else f"{group_name} 합산 {total} ≤ 기준 {limit} — 적합",
                "law_ref": f"{group_threshold['law_source']} {group_threshold['law_article']}"
            })

    has_violations = any(r["status"] == "exceeded" for r in results) or \
                     any(c["status"] == "exceeded" for c in compound_results)

    return {
        "results": results,
        "compound_results": compound_results,
        "has_violations": has_violations
    }
```

```python
from rapidfuzz import process, fuzz
import anthropic

async def recommend_process_codes(
    process_steps: list[str],  # ["Reception", "Founding", "Fermentation", "Distillation", ...]
    original_image_base64: str | None = None
) -> list[dict]:

    # 1단계: Claude API로 공정 키워드 추출 + 설명
    extracted = await extract_process_keywords(process_steps, original_image_base64)
    # extracted = [{"keyword": "발효", "description": "효모를 사용해 당을 에탄올로 전환", "original": "Fermentation"}]

    # 모든 공정 코드 목록 로드 (211개, 서버 시작 시 메모리 캐시)
    all_codes = await get_all_process_codes()  # cache_ttl=3600
    code_names = [(c.code, c.name_ko, c.keywords) for c in all_codes]

    results = []
    for step in extracted:
        keyword = step["keyword"]

        # rapidfuzz 퍼지 매칭: 코드명 + 키워드 배열 대상
        candidates = []
        for code, name, keywords in code_names:
            # 코드명과의 유사도
            name_score = fuzz.token_sort_ratio(keyword, name)
            # 키워드 배열 중 최고 유사도
            kw_score = max((fuzz.token_sort_ratio(keyword, kw) for kw in (keywords or [])), default=0)
            score = max(name_score, kw_score)
            if score > 60:  # 임계값 60점
                candidates.append((code, name, score))

        candidates.sort(key=lambda x: x[2], reverse=True)
        top3 = candidates[:3]

        # Z9(기타) 항상 후보에 포함 (상위 3개가 불명확한 경우)
        if not top3 or top3[0][2] < 75:
            top3.append(("Z9", "기타", 0))

        results.append({
            "process_step": step["original"],
            "keyword_ko": keyword,
            "description": step["description"],
            "candidates": [
                {"code": c[0], "name": c[1], "score": c[2]}
                for c in top3
            ]
        })

    # 근거 설명은 최종 선택 후 1회만 생성 (Claude API 호출 최소화)
    # 매 후보마다 Claude 호출하면 N공정 × 3후보 = 과도한 호출 발생
    # 담당자가 코드 선택 후 confirm 시점에 선택된 코드 1개에 대해서만 근거 생성

    return results
```

**대안 코드 근거 생성**: 담당자가 후보 중 하나를 선택하고 confirm API를 호출하는 시점에 Claude API를 1회 호출해서 "이 공정이 코드 32(분쇄)에 해당하는 이유"를 한 문장으로 생성. 후보마다 미리 생성하면 N공정 × 3후보 = 최대 수십 회 호출 발생. confirm 시점에 1회만 생성해서 `pipeline_steps.edit_reason`에 저장. 식약처 소명 자료로 활용.

**유사 코드 혼동 방지**: `process_codes.similar_codes` 컬럼에 혼동 가능한 코드 명시. 분쇄(32)/으깨기(J11)/짓이김(D4) 선택 시 UI에서 차이점 설명 팝업 표시.

```python
from rapidfuzz import process, fuzz

# 서버 시작 시 12,000개 코드 메모리 로드 (약 5MB, 허용 범위)
_code_cache: list[tuple[str, str]] = []  # [(code, name_ko), ...]

async def match_material_codes(ingredient_name: str) -> list[dict]:
    global _code_cache
    if not _code_cache:
        rows = await db.fetch("SELECT code, name_ko, keywords FROM material_codes")
        _code_cache = [(r.code, r.name_ko) for r in rows]

    # rapidfuzz extractBests: 상위 5개 후보
    matches = process.extractBests(
        ingredient_name,
        [name for _, name in _code_cache],
        scorer=fuzz.token_sort_ratio,
        score_cutoff=70,
        limit=5
    )

    results = []
    for name, score, idx in matches:
        code = _code_cache[idx][0]
        results.append({
            "code": code,
            "name": name,
            "similarity_score": score,
            "note": await get_material_code_note(code)  # DB에서 유사 코드 차이점 조회
        })

    return results[:3]  # 상위 3개만 반환
```

```python
    ingredients = step_results["ingredients"]  # STEP1(원재료 허용 확인) 결과
    food_type = step_results["food_type"]      # STEP2(식품유형 분류) 결과

    # ② 원재료 표기 규칙 DB 조회
    label_rules = await db.query("SELECT * FROM label_rules")

    # ③ 알레르기 대조 (DB vs 원재료 목록)
    allergy_db = await db.query("SELECT * FROM allergy_list")
    detected_allergens = []
    flagged_allergens = []  # 출처 불명확 → 담당자 확인 필요
    for ing in ingredients:
        match = fuzzy_match_allergen(ing["name"], allergy_db)
        if match and ing.get("source_confirmed"):
            detected_allergens.append(match)
        elif match:
            flagged_allergens.append({"ingredient": ing["name"], "reason": "출처 불명확"})

    # ④ 첨가물 표시 의무 조회
    additive_rules = await db.query(
        "SELECT * FROM additive_label_rules WHERE additive_name = ANY($1)",
        [i["name"] for i in ingredients]
    )

    # ⑤ GMO 해당 여부 조회
    gmo_list = await db.query("SELECT * FROM gmo_ingredients")
    gmo_detected = [i for i in ingredients if fuzzy_match(i["name"], gmo_list)]

    # ⑥ 기타 법정 표시사항 — RAG 검색
    rag_context = await pinecone.query(
        f"{step_results['food_type']} 수입식품 한글 표시기준 필수 표시사항",
        top_k=5, filter={"source": "식품등의 표시기준"}
    )

    # 위 데이터 전부 Claude 프롬프트에 주입
    prompt = build_label_prompt(
        food_type=step_results["food_type"],
        ingredients=ingredients,
        label_rules=label_rules,
        detected_allergens=detected_allergens,
        flagged_allergens=flagged_allergens,
        additive_rules=additive_rules,
        gmo_detected=gmo_detected,
        rag_context=rag_context
    )
```

```python
}}
body {{ font-family: 'NanumGothic', sans-serif; font-size: 12px; }}
.header {{ background: #1a365d; color: white; padding: 20px; }}
.section {{ margin: 20px 0; border-bottom: 1px solid #e2e8f0; }}
.pass {{ color: green; }} .fail {{ color: red; }} .warn {{ color: orange; }}
"""

async def generate_report_pdf(case_id: str) -> bytes:
    case = await get_case_with_all_steps(case_id)
    html_content = render_report_template(case)  # Jinja2 템플릿

    pdf_bytes = HTML(string=html_content).write_pdf(
        stylesheets=[CSS(string=REPORT_CSS)]
    )
    return pdf_bytes
```

**폰트 주의사항**: WeasyPrint는 시스템 폰트를 자동 탐색하지 않는다. `NanumGothic.ttf`를 `backend/fonts/` 폴더에 반드시 커밋해야 한다. Railway/Render 배포 시 이 파일이 없으면 한국어가 깨진다. CI에서 확인 체크 추가 권장.

## 5. 비동기 파이프라인 설계

### 5-1. SSE 스트림 구조

5개 기능 파이프라인은 전체 처리에 30~60초 소요 예상. 동기식이면 타임아웃 발생. SSE로 기능별 진행상황을 실시간 전송.

```python
# routers/pipeline.py
from fastapi import APIRouter
from fastapi.responses import StreamingResponse
import asyncio
import json

router = APIRouter()

@router.post("/cases/{case_id}/pipeline/start")
async def start_pipeline(case_id: str, current_user: User = Depends(get_current_user)):
    async def event_generator():
        try:
            async for event in run_pipeline(case_id):
                yield f"data: {json.dumps(event, ensure_ascii=False)}\n\n"
        except Exception as e:
            yield f"data: {json.dumps({'event': 'error', 'message': str(e)})}\n\n"

    return StreamingResponse(
        event_generator(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no"  # Nginx 버퍼링 비활성화 필수
        }
    )
```

### 5-2. 단계별 상태값

```
pending        → 아직 시작 안 됨
running        → Claude API 처리 중
waiting_review → AI 결과 완료, 담당자 확인 대기 중 (여기서 멈춤)
completed      → 담당자 확인 완료
error          → 처리 실패 (재시도 가능)
```

### 5-3. SSE 이벤트 타입 전체

```typescript
// frontend/lib/sse.ts
type PipelineEvent =
  | { event: "feature_start"; feature: number; name: string }
  | { event: "feature_progress"; feature: number; message: string }
  | { event: "feature_result"; feature: number; result: unknown }
  | { event: "feature_waiting"; feature: number; message: "담당자 확인 대기 중" }
  | { event: "feature_confirmed"; feature: number }
  | { event: "feature_error"; feature: number; error: string; retryable: boolean }
  | { event: "pipeline_complete"; case_id: string }
  | { event: "error"; message: string };
```

### 5-4. 중단/재개 처리

각 단계 결과는 `pipeline_steps` 테이블에 즉시 저장. 브라우저 닫아도 진행 상태 유지.

```python
async def run_pipeline(case_id: str):
    steps = await db.fetch("SELECT * FROM pipeline_steps WHERE case_id=$1 ORDER BY step_number", case_id)

    for step_num in range(1, 7):
        step = next((s for s in steps if s.step_number == step_num), None)

        # 이미 완료된 단계는 건너뜀 (재개 처리)
        if step and step.status == "completed":
            yield {"event": "step_confirmed", "step": step_num}
            continue

        # waiting_review 상태면 담당자 확인 대기
        if step and step.status == "waiting_review":
            yield {"event": "step_waiting", "step": step_num, "message": "담당자 확인 대기 중"}
            return  # 스트림 종료, 담당자 confirm 호출 후 재연결

        # 단계 실행
        yield {"event": "step_start", "step": step_num, "name": STEP_NAMES[step_num]}
        try:
            result = await execute_step(case_id, step_num)
            await save_step_result(case_id, step_num, result)
            yield {"event": "step_result", "step": step_num, "result": result}
            yield {"event": "step_waiting", "step": step_num, "message": "담당자 확인 대기 중"}
            return  # 매 단계마다 담당자 확인 후 다음 단계
        except Exception as e:
            await mark_step_error(case_id, step_num, str(e))
            yield {"event": "step_error", "step": step_num, "error": str(e), "retryable": True}
            return
```

## 6. kordoc 통합 방식 결정

### 6-1. 세 가지 선택지 분석

**옵션 A: 마이크로서비스 분리 (Node.js parser-service 별도 배포)**

| | 내용 |
|--|------|
| 구조 | Node.js Express 서버 + `/parse` 엔드포인트. Python 백엔드가 HTTP 호출. |
| 장점 | 언어 선택 자유. Python 팀과 Node 팀 독립 개발. 스케일 분리 가능. |
| 단점 | 서비스 2개 관리. Railway에 2개 프로젝트 배포. 네트워크 레이턴시 추가. |
| 2주 현실성 | 중간. Express 서버 설정이 복잡하지 않음. 인증 토큰만 설정하면 됨. |

**옵션 B: 백엔드를 Node.js + Express로 전환**

| | 내용 |
|--|------|
| 구조 | FastAPI 버리고 Express로 전체 백엔드 재작성. |
| 장점 | kordoc 직접 통합. 언어 통일. |
| 단점 | Python 라이브러리(rapidfuzz, WeasyPrint, Pinecone SDK) 모두 Node 버전으로 교체 필요. 팀이 Python에 익숙하다면 학습 비용 큼. 2주 안에 불가능에 가까움. |
| 2주 현실성 | 낮음. 전체 재작성은 너무 위험. |

**옵션 C: kordoc MCP 서버 활용**

| | 내용 |
|--|------|
| 구조 | kordoc을 MCP 서버로 실행. Claude API가 MCP를 통해 문서 파싱 요청. |
| 장점 | 코드 작성 최소화. Claude가 직접 파일 파싱. |
| 단점 | MCP 서버 운영 복잡도. 프로덕션 환경에서 MCP 안정성 불확실. 배포 방식이 표준적이지 않음. 디버깅 어려움. |
| 2주 현실성 | 낮음. 프로덕션 배포 경험 없는 팀에게는 블랙박스. |

### 6-2. 추천: 옵션 A (마이크로서비스 분리)

2주 데모 기준 옵션 A가 가장 현실적이다.

이유:
1. `parser-service/server.js`는 30줄이면 구현 가능 (`kordoc.convert(filePath)` 래핑)
2. Railway에서 Node.js 서비스 추가 배포는 10분
3. 기존 Python 코드 변경 최소화
4. 나중에 인하우스 서비스로 고도화 가능

실제 구현:

```javascript
// parser-service/server.js
const express = require("express");
const kordoc = require("kordoc");
const multer = require("multer");
const app = express();
const upload = multer({ dest: "uploads/" });

app.post("/parse", upload.single("file"), async (req, res) => {
  // 간단한 토큰 인증
  if (req.headers["x-parser-token"] !== process.env.PARSER_SERVICE_TOKEN) {
    return res.status(401).json({ error: "unauthorized" });
  }

  try {
    const markdown = await kordoc.convert(req.file.path);
    res.json({ markdown, filename: req.file.originalname });
  } catch (err) {
    res.status(500).json({ error: err.message });
  } finally {
    // 임시 파일 정리
    fs.unlink(req.file.path, () => {});
  }
});

app.listen(process.env.PORT || 3001);
```

```python
# services/parser.py (Python 백엔드에서 호출)
async def parse_document_via_kordoc(file_bytes: bytes, filename: str) -> str:
    async with httpx.AsyncClient() as client:
        response = await client.post(
            f"{settings.PARSER_SERVICE_URL}/parse",
            files={"file": (filename, file_bytes)},
            headers={"x-parser-token": settings.PARSER_SERVICE_TOKEN},
            timeout=60.0  # HWP 파싱은 느릴 수 있음
        )
        response.raise_for_status()
        return response.json()["markdown"]
```

## 8. 리스크 및 대응

### 리스크 1: kordoc Node.js 통합

**문제**: Python 백엔드와 언어가 달라 통합 복잡도 존재.
**Plan B**: parser-service 배포 실패 시, HWP 지원을 포기하고 PDF/XLSX만 지원. HWP는 담당자가 먼저 PDF로 변환 후 업로드하도록 안내. 데모에서 HWP가 나오지 않을 가능성도 있음.

### 리스크 2: RAG 청킹 품질

**문제**: 조문 단위 청킹이 잘못되면 "단, 가열제품에 한한다" 같은 조건 절이 유실됨. 기능1 기준규격 판정이 잘못된 결과를 내놓음.
**Plan B**: RAG가 완성되기 전까지 기능1은 Mock thresholds DB만 사용 (핵심 성분 20~30개 수동 입력). 데모에서 RAG 의존도를 줄이고 구조화 DB 중심으로 시연.

### 리스크 3: LLM 임계값 추출 오류

**문제**: LLM이 법령 수치를 잘못 추출하면 DB에 잘못된 기준치 저장 → 기능1 판정 오류.
**대응**: 검수 단계 강제화. `is_verified=false` 인 임계값은 기능1에서 사용 불가, 화면에 "검수 필요" 경고 표시.
**Plan B**: 데모 전날까지 핵심 임계값 20개는 수동으로 직접 입력하고 `is_verified=true`로 설정. LLM 추출은 보조 수단으로만 활용.

### 리스크 4: PDF 한국어 폰트

**문제**: Railway 서버에 한국어 폰트 없으면 PDF가 모두 깨짐. 데모 직전에 발견하면 치명적.
**대응**: `backend/fonts/NanumGothic.ttf` 파일을 git에 커밋. `requirements.txt`에 WeasyPrint 명시. Day 10에 PDF 기능 테스트 시 반드시 실서버에서 확인.
**Plan B**: WeasyPrint 대신 ReportLab + 한국어 폰트 등록. 또는 Puppeteer(Node.js) 기반 PDF 생성으로 전환.

### 리스크 5: 동시접속 편집 충돌

**문제**: 담당자 2~3명이 같은 건을 동시에 편집하면 데이터 충돌.
**대응**: `cases.locked_by` 컬럼 활용. 건 열람 시 잠금 획득, 10분 타임아웃 후 자동 해제. 잠금 중인 건은 다른 담당자에게 "열람만 가능" 안내.
**Plan B**: 데모에서는 단일 사용자만 사용 예정이므로 잠금 로직이 불완전해도 데모에 영향 없음. 실서비스 전에 완성.

### 리스크 6: DeepL 전문 용어 번역 오류

**문제**: "아세틸화", "칼럼흡착" 같은 식품 제조 전문 용어가 DeepL에서 잘못 번역될 수 있음.
**대응**: DeepL 번역 결과를 담당자가 반드시 확인하는 단계 포함. 번역 결과 원문과 함께 표시.
**Plan B**: DeepL 대신 Claude API에서 번역 처리 (기능4 프롬프트에 번역 포함).

### 리스크 7: Claude API 레이턴시

**문제**: 5개 기능 파이프라인에서 Claude API를 4~5회 호출하면 총 처리 시간이 60초 초과 가능.
**대응**: SSE 스트림으로 기능별 진행상황 표시. 사용자는 빈 화면을 기다리지 않음. 각 기능 결과가 나오는 즉시 화면에 표시.
**Plan B**: 기능1(수입가능여부) 내 원재료 허용 대조는 DB 조회만이므로 Claude 호출 없이 빠르게 처리 가능. `asyncio.gather`로 독립적인 DB 조회 병렬 실행.

**문제 3: rapidfuzz 캐싱 전략 누락**

**문제 4: SSE 스트림에서 Nginx 버퍼링 문제 미언급**

**문제 5: parser-service 임시 파일 정리 누락**
- 발견: multer로 업로드된 파일을 파싱 후 삭제하는 코드가 없었음. 서버 디스크 채워질 수 있음.
- 수정: `finally` 블록에 `fs.unlink(req.file.path, () => {})` 추가.

**문제 6: 환경 변수 체크리스트 누락**
