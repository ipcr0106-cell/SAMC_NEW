## 0. 핵심 원칙 (읽기 싫으면 이것만)

1. **내 기능 폴더만 손댄다** — 섹션 2 파일 소유권 참고
2. **공유 파일은 팀 확인 필수** — 혼자 수정하면 다른 기능이 죽음
3. **환경변수에 `F{N}_` prefix 필수** — 안 붙이면 다른 기능과 키 충돌
4. **Supabase 프로젝트 1개만 사용** — 별도 프로젝트 생성 금지
5. **다른 기능 데이터는 `pipeline_steps`에서 조회** — import 금지
6. **프론트 레이아웃은 f0 디자인 유지** — StepNavigation + CaseSummaryPanel + 하단 액션바
7. **`print()` 안에 한글 금지** — Windows에서 서버 죽음

## 2. 파일 소유권 (영역 침범 금지)

### 2-1. 각 기능이 건들 수 있는 범위

| 내 기능 | 건들 수 있는 것 | 절대 금지 |
|---------|---------------|----------|
| 백엔드 | `backend/routers/feature{N}.py`, `backend/services/f{N}_*.py`, `backend/db/feature{N}/` | 다른 기능의 라우터/서비스, `upload.py`, `cases.py`, `main.py` |
| 프론트 | `frontend/features/feature{N}/`, `frontend/app/cases/[id]/f{N}/page.tsx` | 다른 기능의 features 폴더, `dashboard/`, `components/layout/`, `lib/api.ts` |
| DB | `f{N}_` prefix 테이블만 | 다른 prefix 테이블, `cases`, `documents`, `pipeline_steps` 스키마 변경 |
| env | `F{N}_` prefix 키만 | `SUPABASE_URL`, `SUPABASE_SERVICE_KEY` (공용) |
| Pinecone | 자기 기능 인덱스만 | 다른 기능 인덱스 |

### 2-2. 공유 파일 (수정 시 팀 전체 확인 필수)

| 파일 | 왜 위험한가 |
|------|-----------|
| `backend/main.py` | 라우터 등록. 잘못 건드리면 다른 기능 API가 사라짐 |
| `backend/db/supabase_client.py` | 모든 기능의 DB 접속 |
| `backend/requirements.txt` | 패키지 추가/버전 변경은 전체 영향 |
| `frontend/lib/api.ts` | 모든 기능의 API 호출 함수 |
| `frontend/services/apiClient.ts` | axios 인스턴스, baseURL |
| `frontend/types/pipeline.ts` | 기능별 결과 타입 (자기 섹션만 수정) |
| `frontend/components/layout/*` | StepNavigation, CaseSummaryPanel 등 |
| `frontend/app/layout.tsx` | 전역 레이아웃 |
| `frontend/package.json` | 패키지 추가 시 |
| `.env.example` 파일들 | 키 이름 안내 |

### 2-3. 다른 기능의 데이터가 필요할 때


## 5. 커밋 메시지 규칙

```
{type}: {한국어 설명}

type:
  feat     새 기능
  fix      버그 수정
  db       DB 스키마/데이터 변경
  prompt   LLM 프롬프트 수정
  refactor 기능 변화 없는 코드 정리
  chore    설정/패키지/환경변수
  docs     문서
```

금지: `fix: 수정`, `update`, 내용 없는 커밋. 서로 다른 기능을 한 커밋에 묶기.

## 11. 기능 간 데이터 흐름

```
f0 (서류 업로드 + OCR) → pipeline_steps step_key='0'
  ↓
F1 (수입판정) → pipeline_steps step_key='1'
  ↓
F2 (유형분류) → pipeline_steps step_key='2'
  ↓
F3 (필요서류) → pipeline_steps step_key='A'
  ↓
F4 (라벨검토) → pipeline_steps step_key='B'
  ↓
F5 (한글시안) → pipeline_steps step_key='6'
```

각 기능은 이전 기능의 결과를 `pipeline_steps` 테이블에서 조회합니다.
상세 연결 현황은 `MERGE_GUIDE.md`의 "데이터 파이프라인 연결 현황" 참조.
