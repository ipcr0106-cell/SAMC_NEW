## 1. 우리가 만드는 것의 전체 구조

우리 시스템은 크게 세 개의 덩어리로 나뉜다.

**첫 번째: 사용자가 직접 보는 화면 (프론트엔드)**
Vercel이라는 서비스에 올려서 인터넷 주소로 접속하면 바로 뜨는 화면이다. Next.js라는 기술로 만든다.

**두 번째: 실제 계산과 AI 처리를 담당하는 서버 (백엔드)**
사용자가 버튼을 누르면 뒤에서 일하는 서버다. Python FastAPI라는 기술로 만들고, Railway라는 서비스에 올린다. 여기서 Claude(AI), 법령 DB, 기준치 계산이 다 일어난다.

**세 번째: 문서 변환 전용 서버 (파서 서비스)**
HWP, PDF 같은 파일을 AI가 읽을 수 있는 텍스트 형태로 바꿔주는 작업만 전담하는 서버다. kordoc이라는 도구를 쓰는데, 이 도구가 Python이 아닌 Node.js라는 다른 언어로 만들어져 있어서 별도 서버로 분리했다. 이것도 Railway에 올린다.

**왜 셋으로 나눴나?**
사용자 화면 ↔ 서버 ↔ 문서변환서버 이렇게 분리하면, 하나가 고장나도 나머지는 계속 돌아간다. 그리고 팀원들이 각자 담당 부분만 개발할 수 있다.

## 2. 데이터베이스 — 시스템이 기억하는 것들

데이터베이스는 쉽게 말해 **시스템의 기억 창고**다. 우리 시스템은 두 종류의 창고를 쓴다.

### 정형 창고 (Supabase — PostgreSQL)
표 형태로 저장되는 데이터들. 빠르게 숫자를 비교하거나 특정 항목을 찾을 때 쓴다.

### 벡터 창고 (Pinecone)
법령 조문처럼 긴 글을 의미 기반으로 검색할 때 쓴다. "과채음료의 보존료 기준"이라고 검색하면 관련 조문이 자동으로 나오는 방식이다. "RAG"라고 부른다.

---

### 정형 창고에 저장되는 표 목록 (총 14개)

**① cases — 검역 건 목록**
SAMC 담당자가 처리하는 검역 건 하나하나를 기록한다. 제품명, 수입업체명, 현재 상태(처리중/완료/보류/오류), 지금 몇 단계까지 왔는지가 저장된다.
- 추가로, 2~3명의 담당자가 같은 건을 동시에 수정하는 사고를 막기 위해 "잠금" 기능이 있다. 한 사람이 건을 열면 자동으로 잠기고, 10분 동안 아무것도 안 하면 자동으로 잠금이 풀린다.

**② documents — 업로드된 서류 목록**
담당자가 업로드한 PDF, 이미지, HWP 파일 정보가 저장된다. 파일 이름, 어디에 저장됐는지, 파싱(텍스트 추출) 결과, 담당자가 내용을 확인했는지 여부가 기록된다.

**③ f5_pipeline_steps — 8단계 검역 처리 결과**
STEP1(원재료 허용 여부)부터 STEP6(한글 라벨 시안)까지, 중간에 STEP A(수입 필요서류)와 STEP B(수출국 라벨 검토)를 포함한 8단계 전체의 AI 결과와 담당자가 수정한 최종 결과가 저장된다. 담당자가 AI 결과를 왜 바꿨는지 사유도 저장된다 (식약처 소명용).

**④ f5_thresholds — 기준규격 수치 (핵심 중의 핵심)**
"안식향산나트륨은 과채음료에서 0.6g/kg 이하"처럼 성분별 허용 기준치가 저장된다. 이 숫자들은 AI(Claude)가 법령 파일에서 자동으로 뽑아낸다. 단, AI가 숫자를 잘못 읽을 수 있으므로, 사람이 검수(확인)해서 OK 표시를 해야만 실제 검역 판정에 쓸 수 있다. OK 표시가 없는 숫자는 시스템에서 자동으로 사용이 막힌다.
- `created_by` 컬럼: 누가 이 항목을 추가했는지 기록됨. 내가 추가한 것만 수정/삭제 가능.
- `compound_group` 컬럼 (신규): 착색료처럼 여러 종류를 함께 쓸 때 적용되는 "합산 기준치" 그룹을 지정한다. 예: 적색40호와 황색5호가 같은 그룹이면 각각의 기준치 외에 두 성분의 합산값도 별도 기준치와 비교한다.

**⑤ process_codes — 유니패스 제조공정 코드 목록 (211개)**
유니패스에 입력해야 하는 제조공정 코드 전체 목록이다. 각 코드에 한국어 이름, 영문 이름, 매칭에 쓰이는 키워드, 헷갈리기 쉬운 유사 코드 목록이 함께 저장된다.
예: 코드 32는 "분쇄", J11은 "으깨기" — 비슷해 보여도 다른 코드.

**⑥ material_codes — 성분재질 코드 목록 (약 12,000개)**
원재료 각각에 붙이는 성분재질 코드 목록이다. "바닐라 농축액", "바닐라 농축" 같이 거의 같아 보이는 이름이 다른 코드로 존재하는 경우가 많아서 정교한 검색이 필요하다.

**⑦ f5_ingredient_list — 식품원료목록 (포지티브 리스트)**
우리나라에서 식품에 사용할 수 있도록 허용된 원재료 목록이다. 이 목록에 없으면 수입 불가 판정이 난다. 한국어 이름, 학명, 영문명, 사용 부위, 사용 조건이 저장된다.
- `ins_number` / `cas_number` 컬럼 (신규): 해외 라벨에 "Citric Acid (INS 330)"처럼 국제 코드로 성분이 표기된 경우, 이름 매칭이 안 될 때 INS·CAS 번호로 대신 찾을 수 있다. 데이터 출처: 식품첨가물공전 IV.품목별 성분규격 (DB_최신 폴더에 있음).
- `created_by` 컬럼: 담당자가 새 원료를 추가할 수 있고, 자기가 추가한 것만 수정/삭제 가능.

**⑧ f5_law_alerts — 법령 개정 알림 기록**
법령이 바뀌었을 때 이메일 알림을 보낸 기록이 여기 남는다. 어떤 법령이 바뀌었는지, 어디에 영향을 주는지, 이메일이 실제로 발송됐는지가 기록된다.

**⑨ f5_feedback_logs — 담당자 수정 이력**
담당자가 AI 결과를 수정했을 때, 원래 AI가 제안한 것과 담당자가 바꾼 최종값이 모두 저장된다. 나중에 유사한 건이 들어왔을 때 "이전에 비슷한 케이스가 있었어요"라고 참고 자료로 보여줄 수 있게 된다.

**⑩ f5_label_rules — 원재료명 표기 규칙**
"정제수는 % 표기 생략 가능하고 제일 앞에 표기", "카라멜색소는 % 표기 불필요"처럼 라벨 작성 규칙이 저장된다. 법령(`식품등의 표시기준`)에서 가져온 내용이다.
- `created_by` 컬럼: 담당자가 규칙을 추가할 수 있고, 자기가 추가한 것만 수정/삭제 가능.

**⑪ allergy_list — 14대 알레르기 유발 물질 목록**
밀, 대두, 우유, 달걀, 땅콩 등 14가지 알레르기 유발 물질 목록이다. 원재료 중에 이것들이 있으면 라벨에 반드시 표시해야 한다. 동의어(소맥=밀, 글루텐=밀 등)도 함께 저장된다. 법령(`식품등의 표시기준` 별표)이 출처다.
- `created_by` 컬럼 동일하게 적용.

**⑫ additive_f5_label_rules — 첨가물 표시 의무 규칙**
"무수 아황산이 들어간 제품은 라벨에 '이산화황 함유'라고 반드시 써야 한다"처럼, 특정 첨가물이 들어있을 때 라벨에 의무적으로 써야 하는 문구 규칙이 저장된다. 법령(`식품등의 표시기준`)이 출처다.

**⑬ f5_gmo_ingredients — GMO 표시 의무 원료 목록**
대두, 옥수수, 면화 등 유전자변형(GMO) 관련 표시 의무가 있는 원료 목록이다. 함량이 3% 이상이면 라벨에 GMO 관련 표시를 해야 한다. 법령(`유전자변형식품등의 표시기준`)이 출처다.

**⑭ f5_required_documents — 수입 필요서류 목록 (신규)**
식품유형별·조건별로 수입 신고 시 제출해야 하는 서류 목록이다. STEP A에서 이 표를 조회해서 담당자에게 "이 제품은 위생증명서 + OEM 확인서가 필요합니다"처럼 안내해준다.
- 데이터 출처: `DB_최신/6_가이드라인/수입신고시_제출하여야_하는_구비서류_목록(2026.2.5.).xlsx` (이미 폴더에 있음)
- OEM 수입식품, 친환경 동등성 인정(한·미, 한·EU 등) 관련 서류도 포함.

## 3. API — 화면과 서버가 대화하는 방식

API는 화면(프론트엔드)이 서버(백엔드)에게 "이거 해줘"라고 요청하는 통신 방식이다. 모든 요청에는 로그인한 사람이 누구인지 확인하는 인증이 붙는다.

### 검역 건 관련
- 새 건 만들기: 제품명과 수입업체명 입력 → 새 건 생성
- 건 목록 보기: 처리중/완료/보류로 필터, 검색 가능
- 건 상세 보기: 어느 단계까지 왔는지, 각 단계 결과 포함
- 건 상태 바꾸기: 완료 처리, 보류 처리 등
- 편집 잠금 걸기/풀기: 동시 편집 방지용

### 서류 업로드 및 파싱
- 서류 올리기: PDF/이미지/HWP 파일 업로드
- 파싱 시작: 업로드된 파일을 텍스트로 변환 시작 — 진행 상황이 실시간으로 화면에 표시됨
- 파싱 결과 수정: 담당자가 틀린 부분 직접 교정
- 파싱 결과 확인 완료: "내가 확인했다" 표시 — 이걸 눌러야 다음 단계로 넘어감

### 8단계 파이프라인 (순서: STEP1 → STEP2 → STEP A → STEP3 → STEP4 → STEP5 → STEP B → STEP6)
- 파이프라인 시작: 서류 확인 완료 후 AI 8단계 처리 시작
- 특정 단계 재실행: 특정 단계만 다시 돌릴 수 있음 (숫자 단계: 1~6, 알파벳 단계: A, B)
- 단계 결과 수정: AI 결과가 틀렸을 때 담당자가 직접 수정 + 사유 입력
- 단계 확인 완료: 이 단계 OK, 다음 단계로 진행
- 재수입 건 만들기: 이전에 검토한 건을 불러와서 새 건으로 생성. 이전 결과가 초기값으로 채워지고 바뀐 부분만 수정하면 됨 (같은 제조사 제품 재수입 시 활용)

### 법령 관리 (관리자용)
- 법령 파일 업로드: 새 법령 HWP/PDF/엑셀 파일 업로드
- 추출 결과 미리보기: AI가 법령에서 뽑아낸 수치 목록 + 불명확한 항목 표시
- 추출 결과 승인: 내용 확인 후 DB에 반영

### DB 직접 관리 (내가 추가한 항목만)
사용자가 직접 기준치, 알레르기 목록, 첨가물 규칙 등을 추가/수정/삭제할 수 있는 기능이다.
**권한 원칙**: 내가 추가한 항목만 수정/삭제 가능. 시스템이 처음부터 넣어둔 데이터나 다른 사람이 추가한 항목은 수정 불가(조회만 가능).
적용 대상: 기준규격, 알레르기 목록, 첨가물 규칙, GMO 원료, 원재료 표기 규칙, 포지티브 리스트

### PDF 및 이메일
- 검토내역서 PDF 만들기: 6단계 결과 전체를 PDF로 출력
- PDF 다운로드: 만들어진 PDF 내려받기
- 법령 개정 이메일 발송: 법령이 바뀌었을 때 담당자들에게 알림 이메일 보내기

## 4. 기능별 작동 방식 (8단계 파이프라인 상세)

> **순서 변경 안내 (v5)**: 기존 STEP1(식품유형)↔STEP2(원재료) 순서가 바뀌었습니다.
> 회사 요구사항 및 실무자 설문 모두 원재료 허용 여부 확인이 먼저임을 확인.
> 새 순서: STEP1(원재료허용) → STEP2(식품유형) → STEP_A(수입필요서류) → STEP3(기준규격) → STEP4(공정코드) → STEP5(성분코드) → STEP_B(수출국라벨) → STEP6(한글라벨)

### 기능 0 — 법령 DB 파이프라인 (이 모든 것의 기반)

담당자가 식품공전 같은 법령 파일을 업로드하면, 자동으로 다음 순서로 처리된다:

1. 파일 업로드 (HWP/PDF/엑셀)
2. 문서 변환 서버(kordoc)에서 텍스트로 변환
3. 조문 단위로 쪼갬 (중요: 글자 수로 자르면 안 됨. "단, 가열제품에 한한다" 같은 조건 절이 앞 문장과 분리되면 기준이 틀리게 적용됨)
4. 쪼갠 조문들을 벡터 DB(Pinecone)에 저장 → 나중에 "과채음료 보존료 기준" 같은 질문으로 검색 가능
5. 숫자가 있는 법령이면 AI(Claude)가 기준치 수치를 자동 추출
6. 추출된 수치를 사람이 확인 → OK하면 기준규격 DB(f5_thresholds)에 저장 → STEP3에서 사용

**왜 사람이 확인해야 하는가?** AI가 "0.6 g/kg"을 잘못 읽어서 "0.06 g/kg"으로 저장하면 이후 모든 판정이 틀린다. 사람이 반드시 한 번 확인하는 단계를 건너뛸 수 없도록 시스템에서 강제한다.

---

### STEP 1 — 원재료 허용 여부 판정 (수입 가능 여부 1차 판단)

**무엇을 하나**: 원재료 하나하나가 국내 식품에 사용 가능한지 포지티브 리스트(허용 원료 목록)와 대조한다. **수입 불가 원료가 하나라도 있으면 여기서 즉시 중단**되고 이후 단계는 실행하지 않는다.

**왜 제일 먼저 하나**: 수입 자체가 안 되는 제품의 식품 유형을 분류하는 것은 낭비이기 때문이다. 실무자 설문(설문 2차 A1)과 회사 요구사항 모두 이 순서가 맞다고 확인됐다.

**작동 방식**:
1. 서류에서 추출한 원재료 목록의 각 항목을 DB에서 검색
2. 검색 방법 4단계 (앞 단계에서 못 찾으면 다음 단계로):
   - 1순위: 한국어 이름 정확 매칭 ("아가베"라고 검색)
   - 2순위: 학명으로 매칭 ("Agave tequilana"처럼 과학 이름으로 표기된 경우)
   - 3순위: INS 번호 또는 CAS 번호로 매칭 ("Citric Acid (INS 330)"처럼 국제 코드로만 적힌 경우)
   - 4순위: AI(Claude)가 이름을 정규화해서 재검색 (표기 방식이 제각각일 때)
3. DB에 있으면 "허용" 판정 + 근거 법령 표시
4. DB에 없으면 "수입불가 가능성 — 담당자 확인 필요" 경고

**합성향료 특이 처리**:
- "합성향료", "천연향료"라고만 적혀 있으면 어떤 성분인지 알 수 없어서 포지티브 리스트 대조 자체가 불가능하다.
- 이 경우 자동으로 "향료 하위원료 확인 필요 — 거래처에 성분 목록 요청 필요" 경고를 표시한다.
- 참고 법령: 식품첨가물공전 [별표1] (향료에 사용할 수 있는 물질 목록)

---

### STEP 2 — 식품유형 분류

**무엇을 하나**: STEP1에서 원재료 허용 여부가 통과된 후, 이 제품이 식품공전상 어떤 "식품 유형"인지 판단한다. 기준규격(STEP3)은 식품유형에 따라 달라지므로 STEP2가 먼저 필요하다.

**작동 방식**:
1. 우선 "에탄올", "증류", "발효" 같은 키워드가 있는지 빠르게 체크
2. 있으면 주류 가능성으로 표시하고, Claude에게 "주류면 주세법 기준으로 판단해"라고 지시
3. Claude가 원재료 + 제조공정을 분석해서 식품유형 반환
4. 결과 예시: `{ 식품유형: "증류주", 세부유형: "일반증류주", 근거: "제조공정도에 Distillation 확인, 원료가 아가베" }`

**주류 분기 로직**:
- 제조공정에 증류 공정 있음 → 증류주
  - 원료가 보리/맥아 → 위스키
  - 원료가 과실류 → 브랜디
  - 원료가 아가베 → 일반 증류주 (데낄라)
- 증류는 없고 발효 있음 → 발효주 (와인, 맥주 등)
- 주류 아님 → 식품공전 기준 유형 분류

---

### STEP A — 수입 필요서류 안내 (STEP2 직후)

**무엇을 하나**: 식품유형이 결정되면, 이 제품을 수입 신고할 때 어떤 서류를 갖춰야 하는지 자동으로 목록을 뽑아준다.

**왜 이 위치인가**: 필요한 서류가 식품유형에 따라 다르기 때문에, 식품유형이 확정된 직후에 바로 알려주는 것이 실무 흐름과 맞다.

**작동 방식**:
1. STEP2에서 결정된 식품유형으로 구비서류 DB(f5_required_documents 테이블) 조회
2. 추가 조건 확인:
   - 제품명에 수입업체명이 들어가는 OEM 제품이면 → OEM 관련 서류 추가
   - 친환경 유기농 인증 표시가 있으면 → 동등성 인정 서류 추가 (한·미, 한·EU 등)
3. 필수 서류 + 조건부 서류 목록을 화면에 표시
4. 담당자가 목록 보고 준비 여부 체크박스로 표시 가능

**데이터 출처**: DB_최신 폴더에 이미 있는 파일들(수입신고 구비서류 목록 엑셀, OEM 안내서, 동등성 인정 협정문)을 초기 데이터로 입력.

### STEP 3 — 기준규격 적합성 검토 (가장 중요한 숫자 검토)

**무엇을 하나**: 원재료 각각의 함량이 법령에서 정한 허용 기준치를 초과하지 않는지 확인한다.

**가장 중요한 원칙**: AI(Claude)가 직접 계산하지 않는다. DB에서 기준치를 가져오고, 파이썬 코드(if문)가 실제 비교한다. AI한테 "0.5 > 0.6이야?"를 물어보지 않는다는 뜻이다. 컴퓨터 코드로 계산하면 오류가 없다.

**작동 방식**:
1. 원재료 이름 + 식품유형 조합으로 DB에서 기준치 조회
   - 예: "안식향산나트륨" + "과채음료" → 0.6 g/kg (가열제품 한정)
   - **중요**: 사람이 검수 완료한 기준치만 사용. 미검수 항목은 사용 불가.
2. 조건 체크: "가열제품에 한한다"는 조건이 붙은 경우, 해당 제품이 가열 제품인지 확인
3. 단위 변환: g/kg, mg/kg, % 등 단위가 다르면 통일해서 비교
   - 환산계수 처리: 예를 들어 안식향산나트륨은 ×0.847을 해야 안식향산 기준치와 맞춰짐
4. 비교 후 결과:
   - 초과: "실제 0.7 g/kg > 기준 0.6 g/kg — 기준 초과" + 근거 법령
   - 통과: "실제 0.5 g/kg ≤ 기준 0.6 g/kg — 적합"
   - DB에 기준치 없음: "이 성분의 기준치가 DB에 없습니다 — 담당자 직접 확인 필요"
5. **복합 합산 판정** (착색료 등): 같은 그룹의 성분들을 합산해서 그룹 기준치와도 비교
   - 예: 적색40호 실제 0.25 + 황색5호 실제 0.15 = 합산 0.40 > 그룹 기준 0.30 → 합산 초과 판정
   - 각 성분이 개별 기준을 통과해도 합산이 초과하면 최종 불합격

---

### STEP 4 — 제조공정 코드 추천 (유니패스 신고용 코파일럿)

**무엇을 하나**: 제조공정도를 보고 유니패스에 입력할 공정 코드를 추천한다. AI가 직접 입력하지 않는다. 추천만 하고 담당자가 최종 선택.

**작동 방식**:
1. 제조공정도 이미지/텍스트를 Claude에게 주고 "공정 키워드를 한국어로 추출해"라고 요청
2. 추출된 공정 키워드마다, 211개 코드 목록에서 이름이 비슷한 코드 최대 3개를 찾음
   - 예: "발효" → 발효(코드 X), 혐기성발효(코드 Y), 유산발효(코드 Z) 후보 3개 제시
3. 유사도가 낮거나 판단하기 어려우면 "Z9(기타)" 코드도 후보에 자동 포함
4. 담당자가 후보 중 하나를 선택하면 그때 Claude가 "이 코드를 선택한 이유"를 한 문장으로 생성 → 식약처 소명 자료로 활용

**왜 담당자가 최종 선택하나**: 공전에 "세척"과 "세정"의 차이가 명확히 정의되지 않아서 어떤 코드가 맞는지 판단 기준이 없다. 식약처 담당자가 다른 코드를 요구할 수도 있다. 그래서 AI는 근거와 함께 후보를 제시하고, 사람이 책임지고 선택한다.

### STEP 5 — 성분재질 코드 매칭

**무엇을 하나**: 원재료 각각에 맞는 성분재질 코드를 12,000개 목록에서 찾아 추천한다. 이것도 추천만, 담당자가 최종 선택.

**작동 방식**:
- 12,000개 코드를 서버가 켜질 때 메모리에 올려둔다 (매번 DB를 뒤지면 너무 느림)
- "바닐라 농축액"을 검색하면 이름이 비슷한 코드 상위 3개가 나옴
- 왜 글자 유사도로 찾는가: "바닐라 농축"과 "바닐라 농축액"은 의미적으로 같지만 코드가 다르다. AI가 "둘 다 비슷한 의미"라고 판단하면 오히려 헷갈린다. 그냥 글자가 얼마나 비슷한지로 찾는 게 더 정확하다.

### STEP B — 수출국표시사항 검토 (STEP5 직후, STEP6 직전)

**무엇을 하나**: 고객사로부터 받은 해외 원본 라벨 이미지를 AI가 보고, 수입에 부적절한 문구나 그림이 있는지 검토한다. 추가로, 라벨 내용이 앞서 제출한 서류 내용과 일치하는지도 확인한다.

**왜 1순위 기능인가**: 실무자 설문(설문 1차 C1)에서 "가장 먼저 자동화하고 싶은 기능" 1순위로 선택됐다.

**왜 이 위치인가**: STEP1~5에서 원재료·성분이 모두 확정된 후에 라벨과 대조해야 정확하기 때문. 한글 라벨 시안(STEP6)을 만들기 전에 원본 라벨 문제를 먼저 잡아야 한다.

**작동 방식**:
1. 해외 원본 라벨 이미지/PDF를 별도로 업로드
2. 영어 외 언어(중국어, 일본어, 네덜란드어 등)는 자동으로 한국어로 번역 후 분석 (DeepL 번역 API 사용)
3. AI가 부적절한 문구 탐지:
   - 질병 치료·예방 표현 ("혈당 낮춤", "암 예방" 등)
   - 허위·과장 표현 ("100% 천연", "세계 최고 품질" 등)
   - 의약품 오인 표현 ("처방", "치료제" 등)
4. 제품명·원재료(주요 성분)·내용량·원산지·제조사가 이전에 제출한 서류와 일치하는지 교차 확인
5. 결과: 문제 항목 목록 + 라벨 어디에 있는지 설명 + 왜 문제인지 (근거 법령 포함)

### STEP 6 — 한글 라벨 시안 생성

**무엇을 하나**: STEP1~STEP B 결과를 모두 종합해서 한글 라벨 초안을 자동으로 만들어준다.

**중요한 원칙**: AI(Claude)가 법령을 스스로 알고 있다고 믿지 않는다. 모든 판단 기준을 DB에서 가져온 다음 Claude에게 "이 규칙들을 보고 라벨을 써라"고 지시한다. 법령이 바뀌어도 DB만 업데이트하면 AI 결과도 자동으로 바뀐다.

**라벨 항목별 데이터 출처**:

| 라벨 항목 | 어디서 데이터 가져오나 |
|----------|----------------------|
| ① 식품유형 | STEP2(식품유형 분류)에서 판단한 결과 그대로 |
| ② 원재료명 및 함량 표기 규칙 | `f5_label_rules` DB 테이블 (정제수 생략, 카라멜색소 % 불필요 등) |
| ③ 알레르기 유발 물질 14가지 | `allergy_list` DB 테이블 (밀, 대두, 우유 등 14종 + 동의어) |
| ④ 첨가물 표시 의무 | `additive_f5_label_rules` DB 테이블 (무수아황산→이산화황 함유 등) |
| ⑤ GMO 표시 | `f5_gmo_ingredients` DB 테이블 (대두, 옥수수 등) |
| ⑥ 기타 법정 표시사항 | 벡터 DB(Pinecone)에서 관련 법령 조문 검색해서 Claude에게 제공 |

**알레르기 처리 특이사항**:
- "전분"처럼 원료 출처가 명확하지 않으면 자동 확정 안 함
- "밀 전분인지 옥수수 전분인지 불명확 — 담당자 확인 필요"로 표시
- DB에 없는 항목은 Claude가 절대 추측하지 않고 "⚠️ 확인 필요"로 표시

- 시스템에서 DB 업데이트 확인하는 링크

### 기능 12 — 검토내역서 PDF 출력

**무엇을 하나**: 6단계 검역 처리 결과 전체를 A4 PDF 파일로 출력한다. 고객사에 제출하거나 식약처에 소명할 때 쓰는 공식 문서다.

## 5. 처리 도중 화면에 실시간으로 표시되는 방식

6단계를 AI가 처리하는 데 30~60초가 걸린다. 이 동안 화면이 멈춰 있으면 담당자가 고장난 줄 알 수 있다.

그래서 SSE라는 기술을 쓴다. 쉽게 말하면, 서버가 처리 도중에도 중간중간 "지금 1단계 처리 중", "1단계 결과 나왔어요", "담당자 확인 기다리는 중" 같은 메시지를 화면에 실시간으로 보내준다.

각 단계의 상태는 다섯 가지다:
- **대기 중**: 아직 이 단계 안 시작
- **처리 중**: AI가 지금 분석 중
- **담당자 확인 대기**: AI 결과 나왔음, 담당자가 확인 버튼 눌러야 다음으로 넘어감
- **완료**: 담당자가 확인함
- **오류**: 처리 중 에러 발생, 이 단계부터 다시 시작 가능

**중단/재개도 된다**: 브라우저를 닫아도 각 단계 결과가 DB에 저장되어 있어서, 나중에 다시 접속하면 멈춘 단계부터 이어서 진행할 수 있다.

## 6. 문서 파일 변환 방식 선택 이유

**문제**: 담당자가 올리는 서류가 HWP, PDF, 이미지 등 다양하다. 이걸 AI가 읽으려면 텍스트로 변환해야 한다. 특히 HWP 파일은 Python에서 읽는 라이브러리가 제대로 된 게 없다.

**해결책**: kordoc이라는 도구를 쓴다. HWP, HWPX, PDF, 엑셀, 워드를 마크다운(AI가 읽기 좋은 텍스트 형식)으로 변환해준다. 단, 이 도구가 Node.js로 만들어져 있어서 별도 서버로 분리했다.

**세 가지 방법을 검토했다**:
1. **별도 서버로 분리** (선택): Node.js 서버를 따로 하나 만들어서, Python 서버가 여기에 "이 파일 변환해줘"라고 요청하는 방식. 30줄이면 구현 가능.
2. **Python 서버를 Node.js로 전환**: Python을 다 버리고 Node.js로 다시 만드는 것. 빠른 빠른 시간 안에 불가능.
3. **MCP 서버 활용**: 실험적인 방식이라 안정성이 불확실.

→ 2주 데모 기준으로 1번이 가장 현실적이라 선택.

## 8. 예상 위험 요소와 대비책

**위험 1 — 문서 변환 서버(kordoc) 통합 실패**
- 무슨 상황: Node.js 서버와 Python 서버를 연결하는 과정에서 문제가 생길 수 있음
- 대비책: kordoc 통합이 실패하면 HWP 지원을 포기. HWP는 담당자가 먼저 PDF로 변환해서 올리도록 안내. 데모에서 HWP가 나올 가능성이 낮기도 함.

**위험 2 — 법령 조문 청킹 품질 문제**
- 무슨 상황: 법령을 쪼개는 방식이 잘못되면 조건부 기준이 유실됨. "0.6g/kg"이라는 숫자만 남고 "가열제품에 한한다"는 조건이 날아갈 수 있음.
- 대비책: 벡터 DB 완성 전까지는 핵심 성분 20~30개만 수동으로 기준치 DB에 입력해서 사용. 데모에서는 수동 입력 DB 위주로 시연.

**위험 3 — AI가 기준치 수치를 잘못 추출**
- 무슨 상황: AI가 법령에서 "0.6"을 "0.06"으로 잘못 읽으면 모든 판정이 틀림
- 대비책: 사람이 확인해야만 시스템에서 사용 가능하도록 강제. 미확인 항목은 사용 차단.
- 데모 대비책: 발표 전날까지 핵심 기준치 20개를 사람이 직접 입력하고 확인 완료 표시.

**위험 4 — PDF 한국어 깨짐**
- 무슨 상황: 서버에 한국어 폰트가 없으면 PDF에서 모든 한국어가 깨짐. 발표 직전에 발견하면 치명적.
- 대비책: 나눔고딕 폰트 파일을 코드와 함께 서버에 포함. 서버 배포 후 즉시 테스트.

**위험 5 — 여러 담당자가 같은 건을 동시에 수정**
- 무슨 상황: 2~3명이 같은 건을 동시에 열어서 서로 다른 내용을 저장하면 데이터 충돌
- 대비책: 건을 열면 자동으로 편집 잠금 걸림. 다른 사람은 "열람만 가능" 안내.
- 데모 대비책: 데모에서는 한 명만 시연하므로 잠금 기능이 완벽하지 않아도 무방.

**위험 6 — 전문 용어 번역 오류**
- 무슨 상황: DeepL이 "아세틸화", "칼럼흡착" 같은 식품 제조 전문 용어를 틀리게 번역할 수 있음
- 대비책: 번역 결과를 담당자가 반드시 확인. 공정코드 DB에 영문 키워드를 미리 등록해서 번역 없이도 코드 찾기 가능하게 보강.

**위험 7 — AI 처리가 느려서 60초 넘어감**
- 무슨 상황: 6단계에서 AI를 여러 번 쓰면 총 1분 이상 걸릴 수 있음
- 대비책: 단계마다 진행상황을 실시간으로 화면에 표시해서 담당자가 기다리는 동안 답답하지 않게 함.

## 10. 핑퐁 검토에서 수정된 내용 (요약)

개발계획서를 처음 작성한 후 엔지니어 관점에서 3번 검토해서 총 16개 문제를 수정했다.

주요 수정 내용:
1. 동시 편집 잠금 기능 — 처음엔 없었다가 추가
2. 검수 안 된 기준치가 판정에 쓰이는 버그 — SQL 조건 추가로 수정
3. 12,000개 성분재질 코드를 매 요청마다 DB에서 가져오는 성능 문제 — 서버 시작 시 메모리에 올려두는 방식으로 변경
4. SSE가 Nginx 서버에서 버퍼링되는 문제 — 헤더 추가로 해결
5. 문서 변환 후 임시 파일이 서버에 남는 문제 — 삭제 코드 추가
6. 배포 전 체크리스트 없음 — 섹션 9로 추가
7. 공정코드 유사도 기준값 근거 불명확 — 60점 기준 명확화, 75점 미만이면 Z9(기타) 후보 자동 포함
8. AI가 공정코드 후보마다 Claude API를 호출하면 API 비용 폭발 — 담당자가 선택한 시점에 1회만 호출하도록 변경
9. 법령 파일이 1MB 넘으면 DB 저장 성능 이슈 — 별도 저장소에 저장 후 경로만 DB에 기록
10. SAMC 데이터 아직 못 받은 상황 대비 — Mock 데이터 전략 명시
